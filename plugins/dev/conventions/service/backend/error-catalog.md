# 에러 코드 카탈로그

> 버전: v1.0.0
> 최종 수정: 2026-05-19
> 적용 조건: 다음 중 하나라도 해당하면 적용
> - 백엔드가 실패 응답을 반환하는 endpoint를 제공하는 경우 (사실상 모든 백엔드)
> - 클라이언트가 에러 코드로 분기 처리(다국어 메시지, 재시도 결정 등)를 수행하는 경우

---

## 메타-규칙

에러 응답은 **공통 에러 코드 카탈로그**를 기준으로 발급한다. HTTP 상태 코드 → 공통 코드의 매핑 표를 두고, 각 도메인은 자기 도메인의 비즈니스 규칙 위반에 대한 도메인 전용 코드만 추가한다. 도메인 api.md의 에러 표는 본 카탈로그에 존재하는 코드만 참조할 수 있다 — 자체 코드를 즉석에서 만들면 tactical-verifier가 위반으로 검출한다. 에러 응답의 본문 구조는 본 컨벤션이 아닌 `response-envelope.md`에서 결정. 성공 응답의 상태 코드 선택은 `status-codes.md`에서 정하며, rate limit 초과 시의 `429` 코드도 본 카탈로그에 등록한다(`rate-limiting.md`).

---

## 결정 항목

이 컨벤션을 따르려면 다음 항목들을 결정하여 상세설계서에 명시해야 한다.

| 결정 항목 | 명시 위치 | 형태 / 제약 |
|----------|---------|-----------|
| 에러 코드 표기 | `service/{백엔드 서비스}.md` / 에러 카탈로그 섹션 | 단일 케이싱 강제 (보통 UPPER_SNAKE_CASE). 도메인 prefix 부여 정책 결정: ① 항상 prefix (`PROJECT_NOT_FOUND`, `AWS_API_ERROR`) ② 공통 코드는 prefix 없음, 도메인 전용만 prefix |
| HTTP 상태 → 공통 코드 매핑 표 | `service/{백엔드 서비스}.md` / 에러 카탈로그 섹션 | 모든 사용 HTTP 상태 코드에 대한 매핑 명시 (400/401/403/404/409/422/5xx 등). 도메인별 구체 코드를 우선 사용하는 정책도 함께 명시 |
| 공통 에러 코드 카탈로그 | `service/{백엔드 서비스}.md` / 에러 카탈로그 섹션 | 모든 도메인이 공유하는 공통 코드 목록 (예: `VALIDATION_ERROR`, `UNAUTHORIZED`, `FORBIDDEN`, `NOT_FOUND`, `INTERNAL_ERROR`). 코드명 + 의미 + 매핑 HTTP 상태 |
| 도메인별 코드 카탈로그 등록 양식 | `service/{백엔드 서비스}.md` / 에러 카탈로그 섹션 | 각 도메인이 자기 코드를 등록하는 양식. 도메인별 코드는 `service/{백엔드 서비스}.md`에 누적하거나, `domain/{도메인}/api.md`에 두고 본 카탈로그에 도메인 목록만 참조 — 둘 중 위치 정책 결정 |
| 에러 응답 메시지 노출 정책 | `service/{백엔드 서비스}.md` / 에러 카탈로그 섹션 | 둘 중 택일: ① 모든 메시지를 클라이언트에 그대로 노출 ② 일반화된 메시지만 노출, 상세는 서버 로그 (예: `INTERNAL_ERROR`, 외부 API 에러). 일반화 적용 코드 목록 명시 |
| `details` 필드 사용 조건 | `service/{백엔드 서비스}.md` / 에러 카탈로그 섹션 | 어떤 에러 코드에서 `details`를 포함할지 (보통 `VALIDATION_ERROR`의 필드별 오류 배열). 구조도 명시 |

## 상세설계 작성 예

`service/{백엔드 서비스}.md` 에 다음 섹션을 추가한다.

```markdown
## 에러 카탈로그

### HTTP 상태 → 에러 코드 매핑

| HTTP 상태 | 공통 에러 코드 | 도메인 구체 코드 우선 사용 |
|-----------|-------------|-------------------------|
| 400 | `VALIDATION_ERROR` | — (입력 검증은 공통) |
| 401 | `UNAUTHORIZED` | — |
| 403 | `FORBIDDEN` | 권한 세부 사유는 도메인 코드 우선 |
| 404 | `NOT_FOUND` | 리소스 종류별 도메인 코드 우선 (`PROJECT_NOT_FOUND`) |
| 409 | `CONFLICT` | 충돌 사유별 도메인 코드 우선 (`REPO_ALREADY_REGISTERED`) |
| 422 | `UNPROCESSABLE` | 비즈니스 규칙 위반은 도메인 코드 우선 |
| 5xx | `INTERNAL_ERROR` | — (상세는 서버 로그) |

### 공통 에러 코드

| 코드 | HTTP | 의미 |
|------|------|------|
| `VALIDATION_ERROR` | 400 | 입력 검증 실패. `details` 배열에 필드별 오류 포함 |
| `UNAUTHORIZED` | 401 | 인증 필요 또는 토큰 만료 |
| `FORBIDDEN` | 403 | 권한 없음 |
| `NOT_FOUND` | 404 | 리소스 없음 |
| `INTERNAL_ERROR` | 5xx | 서버 내부 오류 (메시지는 일반화) |

### 도메인별 에러 코드

각 도메인의 도메인 전용 코드는 `domain/{도메인}/api.md`의 에러 섹션에 등록하고, 본 카탈로그에는 도메인 목록만 참조한다.

| 도메인 | 코드 목록 위치 |
|-------|---------------|
| project | `domain/project/api.md` |
| approval | `domain/approval/api.md` |
| ... | ... |

### 표기 규칙

- 모든 에러 코드는 UPPER_SNAKE_CASE
- 도메인 전용 코드는 도메인 prefix 강제 (`PROJECT_NOT_FOUND`, `AWS_API_ERROR`)
- 메시지 노출 정책: 비즈니스 규칙 위반은 그대로 노출, `INTERNAL_ERROR`·외부 API 호출 실패는 일반화 메시지만 노출 (원본은 서버 로그)
- `details` 필드는 `VALIDATION_ERROR`에서만 사용 (필드별 `{ field, message }` 배열)
```
