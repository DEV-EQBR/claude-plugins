# HTTP 메서드 시맨틱

> 버전: v1.0.0
> 최종 수정: 2026-06-12
> 적용 조건: 다음 중 하나라도 해당하면 적용
> - 백엔드가 HTTP endpoint를 제공하는 경우 (REST/RPC 스타일 모두 포함)
> - `domain/{도메인}/api.md`에 endpoint의 HTTP 메서드를 명시하는 경우
>
> socket·gRPC 등 비 HTTP 통신은 본 컨벤션의 범위 밖이다.

---

## 메타-규칙

같은 성격의 작업에는 프로젝트 전역에서 **동일한 HTTP 메서드**를 사용한다. 메서드 선택은 임의가 아니라 표준 시맨틱(안전성 safe / 멱등성 idempotent)을 근거로 한다 — 같은 "생성·수정·삭제"에 도메인마다 다른 메서드를 쓰면 클라이언트의 재시도·캐싱 판단이 도메인마다 갈린다. 경로 자체의 표현(액션 endpoint vs 상태 `PATCH`)은 본 컨벤션이 아닌 `api-path.md`에서, 멱등성을 실제로 *보장하는 메커니즘*은 `idempotency.md`에서 결정한다.

> **검증 경로**: 메서드 선택이 service 결정과 도메인 `api.md`에서 일관된지는 Stage 2 LLM judge가 본다. sot-catalog/스크립트의 deterministic 룰은 추가하지 않는다.

## 결정 항목

이 컨벤션을 따르려면 다음 항목들을 결정하여 상세설계서에 명시해야 한다.

| 결정 항목 | 명시 위치 | 형태 / 제약 |
|----------|---------|-----------|
| 생성 메서드 | `service/{백엔드 서비스}.md` / HTTP 메서드 컨벤션 섹션 | 둘 중 택일: ① 항상 `POST` (서버가 ID 생성) ② 클라이언트가 ID를 정하는 경우만 `PUT`. 혼용 시 분기 기준 명시 |
| 수정 방식 (전체 vs 부분) | `service/{백엔드 서비스}.md` / HTTP 메서드 컨벤션 섹션 | `PUT`(전체 교체) 지원 여부 + `PATCH`(부분 수정) 지원 여부. 둘 다 지원 시 사용 기준. 보통 부분 수정 단일화(`PATCH`만) |
| 부분 수정 표현 | `service/{백엔드 서비스}.md` / HTTP 메서드 컨벤션 섹션 | 셋 중 택일: ① 부분 객체(변경 필드만 전송) ② JSON Merge Patch (RFC 7396) ③ JSON Patch (RFC 6902). `null`의 의미(필드 삭제 vs 무시)도 명시 |
| 삭제 시맨틱 | `service/{백엔드 서비스}.md` / HTTP 메서드 컨벤션 섹션 | `DELETE` 사용. 이미 없는 리소스 삭제 시 응답(`204` 멱등 vs `404`) 결정. 하드/소프트 삭제 자체는 도메인 `rules.md` 소관 |
| 조회 메서드 | `service/{백엔드 서비스}.md` / HTTP 메서드 컨벤션 섹션 | `GET` 강제(safe·idempotent). `GET`에 요청 본문 금지 |
| 긴/민감 조회 처리 | `service/{백엔드 서비스}.md` / HTTP 메서드 컨벤션 섹션 | 쿼리가 길거나 민감 정보를 담는 검색을 `GET`(쿼리스트링)로 유지할지 `POST`(검색 body)로 둘지 기준. `POST` 검색은 비안전·비멱등·캐싱 대상 아님을 명시 |
| 메서드별 성질 선언 | `service/{백엔드 서비스}.md` / HTTP 메서드 컨벤션 섹션 | 사용 메서드별 safe/idempotent 표를 명시하여 클라이언트 재시도 정책의 근거 제공 (`idempotency.md`·`caching.md`가 참조) |

## 상세설계 작성 예

`service/{백엔드 서비스}.md` 에 다음 섹션을 추가한다.

```markdown
## HTTP 메서드 컨벤션

| 작업 | 메서드 | safe | idempotent | 비고 |
|------|--------|------|-----------|------|
| 조회 | `GET` | ✅ | ✅ | 요청 본문 금지 |
| 생성 | `POST` | ❌ | ❌ | 서버가 ID 생성. 멱등 필요 시 `idempotency.md`의 키 적용 |
| 부분 수정 | `PATCH` | ❌ | ✅ | 부분 객체 전송. `null`은 필드 삭제 의미 |
| 삭제 | `DELETE` | ❌ | ✅ | 이미 없으면 `204`로 멱등 처리 |

- 전체 교체(`PUT`)는 사용하지 않는다 — 수정은 `PATCH`로 단일화
- 긴 조건 검색은 `POST /{resource}/search` 사용 (비안전·비멱등, 캐싱 대상 아님)
```
