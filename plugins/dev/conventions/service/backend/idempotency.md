# 멱등성 보장

> 버전: v1.0.0
> 최종 수정: 2026-06-12
> 적용 조건: 다음 중 하나라도 해당하면 적용
> - 비멱등 메서드(`POST`)로 부수효과(생성·결제·발송 등)를 일으키는 endpoint가 있는 경우
> - 클라이언트 재시도·네트워크 불안정으로 중복 요청이 발생할 수 있는 경우
>
> socket·gRPC 등 비 HTTP 통신은 범위 밖이다.

---

## 메타-규칙

재시도해도 안전해야 하는 **비멱등 작업(주로 `POST` 생성·결제)에는 멱등성 키 메커니즘으로 중복 실행을 방지**한다. `GET`/`PUT`/`DELETE`의 표준 멱등 성질은 `http-methods.md`가 선언하고, 본 컨벤션은 그렇지 못한 작업에 멱등성을 *부여하는 방법*을 정한다. 중복으로 판정된 재요청은 같은 결과를 반환해야 하며, 그 응답 본문은 `response-envelope.md` 구조를, 거부 시 에러 코드는 `error-catalog.md` 카탈로그를 따른다.

> **검증 경로**: 이 컨벤션의 *동작*(같은 키 재요청 시 1회만 실행)은 정적 상세설계 검증으로 잡히지 않는다 — fragment의 "검증 항목"에 기록하여 런타임 `verifier`가 확인한다.

## 결정 항목

이 컨벤션을 따르려면 다음 항목들을 결정하여 상세설계서에 명시해야 한다.

| 결정 항목 | 명시 위치 | 형태 / 제약 |
|----------|---------|-----------|
| 멱등성 키 적용 대상 | `service/{백엔드 서비스}.md` / 멱등성 섹션 | 어느 endpoint에 키를 요구/허용할지. 전체 `POST` 강제 vs 선별(결제·주문 등 부수효과 큰 작업). 도메인별 추가 대상은 `api.md`에서 표기 |
| 키 전달 방식 | `service/{백엔드 서비스}.md` / 멱등성 섹션 | 헤더명 단일화(보통 `Idempotency-Key`). 값 형식(클라이언트 생성 UUID 등)과 검증 규칙 |
| 키 보존 기간 (window) | `service/{백엔드 서비스}.md` / 멱등성 섹션 | 서버가 키↔결과를 기억하는 기간(예: 24h). 만료 후 같은 키 재사용 시 동작 |
| 중복 요청 처리 | `service/{백엔드 서비스}.md` / 멱등성 섹션 | 완료된 키 재요청 시 저장된 원응답을 그대로 반환. 처리 중(in-flight)이면 `409` vs 대기 중 택일 |
| 키 충돌 처리 | `service/{백엔드 서비스}.md` / 멱등성 섹션 | 같은 키 + 다른 요청 본문일 때 거부 여부(`422`/`409`). 거부 코드는 `error-catalog.md` |
| 키 누락 처리 | `service/{백엔드 서비스}.md` / 멱등성 섹션 | 키 필수 endpoint에서 누락 시: `400` 거부 vs 멱등 보장 없이 진행 중 택일 |
| 키 스코프 | `service/{백엔드 서비스}.md` / 멱등성 섹션 | 키 유효 범위: 사용자별 vs 전역. 다른 사용자가 같은 키를 보낼 때의 격리 |

## 상세설계 작성 예

`service/{백엔드 서비스}.md` 에 다음 섹션을 추가한다.

```markdown
## 멱등성

| 항목 | 정책 |
|------|------|
| 적용 대상 | 결제·주문 생성 등 부수효과 큰 `POST`. 도메인별 대상은 각 `api.md`에 표기 |
| 키 전달 | `Idempotency-Key` 헤더 (클라이언트 생성 UUIDv4) |
| 보존 기간 | 24시간. 만료 후 동일 키는 새 요청으로 처리 |
| 중복 처리 | 완료 키 → 저장된 원응답 반환. 처리 중 → `409 CONFLICT` |
| 키 충돌 | 같은 키·다른 본문 → `422 UNPROCESSABLE` |
| 키 누락 | 적용 대상 endpoint에서 누락 → `400 VALIDATION_ERROR` |
| 키 스코프 | 사용자별 (인증 주체 + 키 조합) |
```
