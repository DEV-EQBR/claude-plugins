# Rate Limiting

> 버전: v1.0.0
> 최종 수정: 2026-06-12
> 적용 조건: 다음 중 하나라도 해당하면 적용
> - 외부/공개에 노출되는 HTTP API가 있는 경우
> - 남용·과부하로부터 보호가 필요하거나, 사용량 기반 정책이 있는 경우
>
> socket·gRPC 등 비 HTTP 통신은 범위 밖이다.

---

## 메타-규칙

요청 빈도 제한의 **기준(식별 단위 · 임계값 등급 · 초과 응답 형식)을 전역에서 일관되게** 정한다. 도메인마다 다른 헤더·다른 초과 응답을 쓰지 않는다. 초과 시 반환하는 코드는 `error-catalog.md`의 `429` 매핑을, 응답 본문은 `response-envelope.md`를 따른다. 임계값의 구체 수치는 비기능 요구사항이므로 보안·용량 정책과 함께 결정된다(`L3` 가능).

> **검증 경로**: 이 컨벤션의 *동작*(임계 초과 시 차단)은 부하에 의존하므로 정적 검증 대상이 아니다 — fragment "검증 항목"에 기록하여 런타임 `verifier`가 가능한 범위에서 확인한다(초과 호출 → `429` + 헤더).

## 결정 항목

이 컨벤션을 따르려면 다음 항목들을 결정하여 명세서에 명시해야 한다.

| 결정 항목 | 명시 위치 | 형태 / 제약 |
|----------|---------|-----------|
| 식별 단위 | `service/{백엔드 서비스}.md` / Rate Limit 섹션 | 무엇을 기준으로 셀지: IP / 인증 사용자 / API 키 / 조합. 미인증·인증 요청의 단위 구분 |
| 임계값 등급 | `service/{백엔드 서비스}.md` / Rate Limit 섹션 | 전역 단일 vs endpoint 등급별(읽기 / 쓰기 / 비용 큰 작업). 각 등급의 한도와 윈도우(고정 vs 슬라이딩). 구체 수치는 비기능 요구사항 |
| 초과 응답 | `service/{백엔드 서비스}.md` / Rate Limit 섹션 | `429 Too Many Requests` + 본문(`error-catalog.md` 코드) + `Retry-After` 헤더 포함 여부 |
| 한도 정보 헤더 | `service/{백엔드 서비스}.md` / Rate Limit 섹션 | 잔여 한도 노출 헤더 사용 여부·형식: 표준 `RateLimit-*` vs `X-RateLimit-*`. 노출 항목(Limit/Remaining/Reset) |
| 면제 / 화이트리스트 | `service/{백엔드 서비스}.md` / Rate Limit 섹션 | 내부 서비스·신뢰 클라이언트 면제 정책. 면제 식별 방법 |
| 적용 계층 | `service/{백엔드 서비스}.md` / Rate Limit 섹션 | 어디서 적용하는지(게이트웨이 / 미들웨어). 분산 환경에서 카운트 일관성 보장 방식(정책 수준 명시) |

## spec 작성 예

`service/{백엔드 서비스}.md` 에 다음 섹션을 추가한다.

```markdown
## Rate Limit

| 항목 | 정책 |
|------|------|
| 식별 단위 | 인증 요청=사용자 ID, 미인증 요청=IP |
| 임계값 등급 | 읽기 600/분, 쓰기 120/분, 비용 큰 작업(리포트 등) 10/분. 슬라이딩 윈도우 |
| 초과 응답 | `429` + `{ error: { code: "RATE_LIMITED" } }` + `Retry-After: <초>` |
| 한도 헤더 | `RateLimit-Limit` / `RateLimit-Remaining` / `RateLimit-Reset` 노출 |
| 면제 | 내부 서비스 토큰은 면제 (게이트웨이에서 식별) |
| 적용 계층 | API 게이트웨이. 카운터는 공유 저장소로 분산 일관성 확보 |

> `RATE_LIMITED` 코드는 `error-catalog.md`의 `429` 매핑에 등록한다.
```
