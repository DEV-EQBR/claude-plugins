# 리소스 표현 (관계·부분 응답)

> 버전: v1.0.0
> 최종 수정: 2026-06-12
> 적용 조건: 다음 중 하나라도 해당하면 적용
> - 백엔드가 HTTP endpoint로 JSON 리소스를 반환하는 경우
> - 한 리소스가 다른 도메인/엔티티를 참조하여 응답에 함께 표현해야 하는 경우
>
> socket·gRPC 등 비 HTTP 통신은 범위 밖이다.

---

## 메타-규칙

한 리소스가 다른 리소스를 참조할 때, 응답에 그것을 **어떻게 표현할지(중첩 임베드 vs 참조 ID)를 프로젝트 전역의 일관된 기본값**으로 정한다. 같은 엔티티가 단일 조회·목록·임베드 위치에서 서로 다른 필드셋으로 표류하지 않게 한다. 응답 본문의 공통 래퍼(`success`/`data`)는 `response-envelope.md`, 컬렉션 구조(`items`/`pagination`)는 `list-response.md`가 결정한다 — 본 컨벤션은 그 `data` 안에 담기는 **리소스 본문의 표현 형태**만 결정한다. 필드의 이름·케이싱은 `field-naming.md` 소관이다.

> **검증 경로**: 관계 표현 기본값·요약/상세 구분이 명세에 일관되게 박혔는지는 Stage 2 LLM judge가 본다. deterministic 룰은 추가하지 않는다.

---

## 결정 항목

이 컨벤션을 따르려면 다음 항목들을 결정하여 명세서에 명시해야 한다.

| 결정 항목 | 명시 위치 | 형태 / 제약 |
|----------|---------|-----------|
| 관계 표현 기본값 | `service/{백엔드 서비스}.md` / 리소스 표현 섹션 | 셋 중 택일: ① 참조 ID만 (`authorId`) ② 요약 객체 임베드 (`author: { id, name }`) ③ 전체 임베드. 도메인 간 혼재 금지. 기본값 + 예외 허용 기준 명시 |
| expand / include 지원 | `service/{백엔드 서비스}.md` / 리소스 표현 섹션 | 클라이언트가 관계를 펼치도록 요청하는 기능 지원 여부. 지원 시: 파라미터명(`expand` vs `include`)·허용 관계 화이트리스트·중첩 깊이 상한 |
| 부분 응답 (sparse fieldset) | `service/{백엔드 서비스}.md` / 리소스 표현 섹션 | 필드 선택 지원 여부. 지원 시: 파라미터명(보통 `fields`)·문법(쉼표 구분)·미전송 시 기본 동작(전체 반환). webview·저대역 클라이언트 대비 고려 |
| 요약 vs 상세 표현 | `service/{백엔드 서비스}.md` / 리소스 표현 섹션 | 같은 엔티티의 목록용(summary)과 단일 조회용(detail) 필드셋 분리 여부. 분리 시 각 표현이 담는 필드 경계 결정. 임베드 시 어느 표현을 쓸지도 명시 |
| 중첩 깊이 / 순환 처리 | `service/{백엔드 서비스}.md` / 리소스 표현 섹션 | 임베드 최대 깊이. 순환 참조(A→B→A) 차단 방식(깊이 제한 / ID로 절단) 결정 |
| 대용량·민감 필드 기본 포함 | `service/{백엔드 서비스}.md` / 리소스 표현 섹션 | 무거운 필드(긴 본문·바이너리)·민감 필드를 기본 응답에 포함할지, expand/fields로만 노출할지. 민감 필드 노출 정책 자체는 `rules.md` 소관 |
| 관계 없음 표현 | `service/{백엔드 서비스}.md` / 리소스 표현 섹션 | 참조가 없을 때: `null` 명시 vs 키 생략. 단일 정책으로 통일 (`field-naming.md`의 boolean/optional 정책과 정합) |

## spec 작성 예

`service/{백엔드 서비스}.md` 에 다음 섹션을 추가한다.

```markdown
## 리소스 표현

| 항목 | 정책 |
|------|------|
| 관계 표현 기본값 | 요약 객체 임베드 (`author: { id, name }`). 1:N의 N측은 참조 ID만 |
| expand 지원 | `?expand=` 지원. 허용: `author`, `project`. 깊이 1단계까지 |
| 부분 응답 | `?fields=` 지원 (쉼표 구분). 미전송 시 기본 필드셋 전체 반환 |
| 요약 vs 상세 | 목록=summary(`id, title, status`), 단일 조회=detail(전체). 임베드는 summary 사용 |
| 중첩 깊이 | 최대 1단계. 순환 참조는 ID로 절단 |
| 대용량 필드 | `content`(본문)는 단일 조회에만 포함, 목록 응답에서 제외 |
| 관계 없음 | `null` 명시 (키 생략 금지) |

### 표현 예

\`\`\`
GET /api/v1/posts/42
{ "success": true, "data": {
  "id": 42, "title": "...", "author": { "id": 7, "name": "Kim" }, "projectId": 3
} }

GET /api/v1/posts/42?expand=project&fields=id,title,project
{ "success": true, "data": {
  "id": 42, "title": "...", "project": { "id": 3, "name": "..." }
} }
\`\`\`
```
