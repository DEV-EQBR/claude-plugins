# 컴포넌트 파운데이션 (UI 기반·베이스라인 컴포넌트 세트)

> 버전: v1.0.0
> 최종 수정: 2026-06-26
> 적용 조건: 다음에 해당하면 적용
> - 화면(`screens.md`)이 정의된 모든 프론트엔드 서비스 (플랫폼 무관 — web·네이티브 공통)

---

## 메타-규칙

UI 프리미티브(버튼·입력·선택·다이얼로그 등)를 화면마다 바닥부터 새로 만들지 않는다. 프로젝트는 **하나의 컴포넌트 파운데이션**을 정하고 모든 화면이 그 위에서 조립된다. 파운데이션은 접근성·상태·키보드 동작이 이미 검증된 것을 고른다 — 직접 구현하면 매번 hover/focus/disabled/a11y를 놓쳐 품질이 무너진다. 같은 종류의 요소(예: 모든 1차 버튼)는 화면이 달라도 같은 컴포넌트를 쓴다 — 일관성은 재사용에서 나온다.

> 이 컨벤션은 "검증된 파운데이션 위에서 조립한다"는 *원칙*과 *어떤 컴포넌트가 베이스라인에 있어야 하는가*를 강제한다. *어떤 파운데이션을 고르는가*는 프로젝트의 기술 스택에 묶인 결정이다(기존 코드베이스가 있으면 그 선택을 우선한다).

## 결정 항목

이 컨벤션을 따르려면 다음 항목들을 결정하여 명세서에 명시해야 한다.

| 결정 항목 | 명시 위치 | 형태 / 제약 |
|----------|---------|-----------|
| 파운데이션 방식 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 셋 중 택일: ① 헤드리스+스타일(예: Radix/Headless UI 등에 토큰 스타일 적용) ② 완성 킷(예: 디자인 시스템 라이브러리 채택) ③ 자체 디자인 시스템(직접 구축 — 베이스라인 세트를 모두 자작). 기존 코드베이스가 있으면 그 방식을 따른다 |
| 파운데이션 식별 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 채택한 라이브러리/프레임워크 이름과 버전 라인. 자체 구축이면 "자체"로 명시하고 컴포넌트 소스 위치 |
| 베이스라인 컴포넌트 세트 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 화면들이 공통으로 쓰는 컴포넌트 목록. 최소 권장: Button(1차/2차/위험) · Input/Textarea · Select · Checkbox/Radio · Card · Dialog/Modal · Toast/Alert · Table/List · Tabs · Badge/Chip · Spinner/Skeleton. 화면에 필요 없는 것은 제외 가능 |
| 토큰 결합 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 파운데이션이 `design-tokens.md`의 토큰을 어떻게 소비하는가(테마 설정·CSS 변수·테마 프로바이더 등 메커니즘 한 줄) |

## spec 작성 예

`service/{프론트엔드 서비스}.md` 에 다음 섹션을 추가한다.

```markdown
## 디자인 정책 - 컴포넌트 파운데이션

| 항목 | 정책 |
|------|------|
| 파운데이션 방식 | 헤드리스+스타일 (동작·a11y는 헤드리스 프리미티브, 시각은 디자인 토큰으로 직접 스타일) |
| 파운데이션 식별 | Radix Primitives + Tailwind (토큰을 Tailwind theme에 매핑) |
| 베이스라인 컴포넌트 세트 | Button(primary/secondary/danger) · Input · Textarea · Select · Checkbox · Radio · Card · Dialog · Toast · Table · Tabs · Badge · Spinner · Skeleton |
| 토큰 결합 | 디자인 토큰을 CSS 변수로 노출 → Tailwind theme이 참조. 다크모드는 `data-theme` 속성 토글 |
```
