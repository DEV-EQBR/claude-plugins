# 컴포넌트·화면 상태 커버리지

> 버전: v1.0.0
> 최종 수정: 2026-06-26
> 적용 조건: 다음에 해당하면 적용
> - 화면(`screens.md`)이 정의된 모든 프론트엔드 서비스 (플랫폼 무관 — web·네이티브 공통)

---

## 메타-규칙

화면과 컴포넌트는 "정상 데이터가 채워진 상태" 하나만으로 끝나지 않는다. 사용자가 실제로 만나는 **모든 상태가 설계되어야** 한다 — 빈 상태·로딩·에러·부분 실패·권한 없음. 상태 누락이 체감 품질 저하의 가장 큰 원인이다(빈 목록이 그냥 흰 화면, 로딩 중 레이아웃 점프, 에러 시 멈춤). 인터랙티브 요소는 시각적으로 자기 상태를 드러내야 한다 — 누를 수 있는지(hover), 포커스됐는지(focus), 비활성인지(disabled), 처리 중인지(loading)를 사용자가 알 수 있어야 한다.

> 이 컨벤션은 *어떤 상태가 빠짐없이 다뤄져야 하는가*를 강제한다. 각 상태의 *구체 표현*(빈 상태 문구·일러스트 등)은 화면별로 `ui.md`에 설계된다.

---

## 결정 항목

이 컨벤션을 따르려면 다음 항목들을 결정하여 상세설계서에 명시해야 한다.

| 결정 항목 | 명시 위치 | 형태 / 제약 |
|----------|---------|-----------|
| 인터랙티브 상태 세트 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 모든 인터랙티브 컴포넌트가 표현해야 할 상태: `default`·`hover`(포인터 환경)·`focus-visible`·`active`·`disabled`·`loading`(비동기 트리거 시). 각 상태의 표현 기준 |
| 데이터 표시 상태 세트 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 데이터를 보여주는 모든 영역이 다뤄야 할 상태: `loading`(스켈레톤/스피너 기준) · `empty`(데이터 0건) · `error`(로드 실패 + 재시도) · `partial`(일부 실패/페이지네이션). 적용 기준 |
| 빈 상태 기본 패턴 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 빈 상태의 표준 구성(안내 문구 + 다음 행동 유도). "그냥 빈 화면" 금지 |
| 로딩 표현 기준 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 셋 중: ① 스켈레톤(레이아웃 보존) ② 스피너 ③ 혼용 기준. 레이아웃 점프(CLS) 방지 원칙 |
| 폼 검증 피드백 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | `screens.md`/`ui.md`의 폼 검증 결과를 어떻게 표현하는가(인라인 에러 위치·시점: blur/submit). 에러 메시지 톤은 도메인 상세설계 따름 |

## 상세설계 작성 예

`service/{프론트엔드 서비스}.md` 에 다음 섹션을 추가한다.

```markdown
## 디자인 정책 - 상태 커버리지

### 인터랙티브 상태 (모든 버튼/입력/링크 의무)
default · hover(포인터 환경만) · focus-visible(키보드 포커스 링 의무) · active · disabled(40% 불투명 + 포인터 차단) · loading(비동기 액션 시 스피너 + 중복 제출 차단)

### 데이터 표시 상태 (모든 목록/상세 영역 의무)
| 상태 | 표현 |
|------|------|
| loading | 스켈레톤(실제 레이아웃과 동형 — 점프 방지) |
| empty | 안내 문구 + 1차 행동 버튼 ("아직 주문이 없어요" + [주문하기]) |
| error | 에러 메시지 + [다시 시도] 버튼 (멈춤/흰 화면 금지) |
| partial | 다음 페이지 로딩 인디케이터 / 일부 항목 실패 표시 |

### 폼 검증 피드백
인라인(필드 하단), blur 시 1차 검증 + submit 시 전체 검증. 에러 필드로 포커스 이동.
```
