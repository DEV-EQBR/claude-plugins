# 전화번호 입력·표시

> 버전: v1.0.0
> 최종 수정: 2026-06-09
> 적용 조건: 다음 중 하나라도 해당하면 적용
> - 화면(`screens.md`)에 전화번호 입력 요소가 있는 경우
> - 화면에 전화번호 표시 요소가 있는 경우
> - 도메인 엔티티(`entities.md`)에 전화번호 속성이 있는 경우

---

## 메타-규칙

전화번호는 국가별 형식 차이를 단일 정책으로 흡수한다. **저장은 E.164 정규형(국가코드 포함)** 단일 표준으로 통일하고, **입력 검증과 표시**는 국가별 규칙을 적용한다. 국가별 검증 규칙(정규식)은 단일 출처에서 관리하며, 화면마다 임의로 패턴을 정의하지 않는다.

---

## 결정 항목

이 컨벤션을 따르려면 다음 항목들을 결정하여 명세서에 명시해야 한다.

| 결정 항목 | 명시 위치 | 형태 / 제약 |
|----------|---------|-----------|
| 지원 국가 목록 | `service/{프론트엔드 서비스}.md` / 입력 형식 섹션 | 지원 국가 + 국가 코드(`+82` 등). 단일 국가만 지원하면 그 국가 명시 |
| 국가별 검증 정규식 | `service/{프론트엔드 서비스}.md` / 입력 형식 섹션 | 국가별 패턴. 직접 정규식을 박거나 단일 소스(상수 테이블/라이브러리, 예: `libphonenumber`) 명시 |
| 저장 정규형 | `system.md` / 시간·식별자 등 데이터 표준 섹션 (또는 `service/{백엔드 서비스}.md` / 필드 명명) | E.164 권장 (예: `+821012345678`). 백엔드 검증도 동일 규칙 미러링 |
| 표시 형식 | `service/{프론트엔드 서비스}.md` / 표시 형식 섹션 | 국가별 하이픈/괄호 규칙 (예: 한국 `010-1234-5678`). 저장 정규형 → 표시 변환은 프론트엔드 단일 경계에서 수행 |

> 정규식 검증은 프론트엔드 입력 시점뿐 아니라 백엔드 생성 시점에서도 미러링되어야 한다. 저장 정규형 결정은 그래서 system/backend 차원에 박힌다.

## spec 작성 예

`service/{프론트엔드 서비스}.md` 에 다음 섹션을 추가한다.

```markdown
## 입력 형식 - 전화번호

| 국가 | 국가 코드 | 검증 정규식 (소스) | 표시 형식 |
|------|---------|------------------|----------|
| 한국 | +82 | `^01[016789]\d{7,8}$` (상수 테이블) | `010-1234-5678` |
| 미국 | +1 | `^\d{10}$` (상수 테이블) | `(123) 456-7890` |

> 검증 규칙은 단일 상수 테이블에서 관리. 화면별 임의 정의 금지.
```

`system.md` (또는 `service/{백엔드 서비스}.md`) 에는 저장 정규형을 박는다.

```markdown
## 데이터 표준 - 전화번호

| 항목 | 정책 |
|------|------|
| 저장 정규형 | E.164 (`+821012345678`). 국가 코드 포함, 구분자 없음 |
| 검증 | 백엔드 생성 시 프론트엔드와 동일한 국가별 규칙 미러링 |
```
