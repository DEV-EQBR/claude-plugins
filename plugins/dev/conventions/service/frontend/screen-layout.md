> 버전: v1.0.0
> 최종 수정: 2026-06-23
> 적용 조건: 다음에 해당하면 적용
> - 화면(`screens.md`)이 정의된 모든 프론트엔드 서비스 (플랫폼 무관 — web·네이티브 공통)

---

## 메타-규칙

화면 레이아웃은 OS/플랫폼(web·iOS·Android)이 아니라 **폼팩터(데스크톱·태블릿·모바일)**를 기준으로 결정한다. 같은 폼팩터는 플랫폼이 달라도 같은 레이아웃 관심사를 공유한다 — "모바일 레이아웃"은 모바일 웹이든 네이티브 앱이든 동일한 문제(좁은 너비·터치·세로 우선)다. 프로젝트는 **지원 폼팩터**를 정하고, 폼팩터별 기준 크기와 시스템 예약 영역·사용자 글자 확대·화면 회전 대응을 명시한다. 지원 범위 밖 동작도 명시하여 예측 불가한 깨짐을 차단한다.

> 빌드·검증·배포 *도구*(브라우저 자동화 vs 네이티브 시뮬레이터 등)는 폼팩터가 아니라 런타임 종류에 따라 갈린다 — 그건 이 컨벤션이 아니라 `system/local-verification.md`의 "배포 타깃 종류 / UI 검증 드라이버"가 정한다. 여기서는 **레이아웃 규칙**만 다룬다.

## 결정 항목

이 컨벤션을 따르려면 다음 항목들을 결정하여 명세서에 명시해야 한다.

| 결정 항목 | 명시 위치 | 형태 / 제약 |
|----------|---------|-----------|
| 지원 폼팩터 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | `데스크톱` / `태블릿` / `모바일` 중 1개 이상 (예: "모바일 전용", "데스크톱 + 모바일") |
| 폼팩터별 기준 크기 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 폼팩터별 기준/최소 크기. web = px 브레이크포인트(예: 모바일 360–767, 태블릿 768–1023, 데스크톱 1024+), 네이티브 = 기준 기기군(예: 모바일 = iPhone, 태블릿 = iPad) |
| 시스템 예약 영역 (safe area) | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 콘텐츠가 침범하면 안 되는 영역 처리 원칙 — 노치·상태바·홈 인디케이터(네이티브) 또는 브라우저 UI·노치(모바일 웹). 예: "콘텐츠는 safe area 내부, 배경만 edge-to-edge" |
| 사용자 글자 확대 대응 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | OS 글자 크기 설정(네이티브 Dynamic Type)·브라우저 확대 시 레이아웃 유지 범위. 지원 범위 + 초과 시 처리(클램프/스크롤) |
| Orientation | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 셋 중 택일: ① 세로 고정 ② 가로 고정 ③ 회전 지원. 회전 지원 시 레이아웃 적응 방식 명시 |
| 지원 범위 밖 동작 | `service/{프론트엔드 서비스}.md` / 디자인 정책 섹션 | 미지원 폼팩터/크기에서의 동작: ① 가로 스크롤 허용 ② 안내 메시지 ③ 접근 차단 |

## spec 작성 예

`service/{프론트엔드 서비스}.md` 에 다음 섹션을 추가한다.

### 데스크톱 + 모바일 (web)

```markdown
## 디자인 정책 - 화면 레이아웃

| 항목 | 정책 |
|------|------|
| 지원 폼팩터 | 데스크톱 + 모바일 (태블릿은 모바일 레이아웃으로 흡수) |
| 폼팩터별 기준 크기 | 모바일 360–767px, 데스크톱 1024px+ (768–1023은 모바일 레이아웃 확장) |
| 시스템 예약 영역 | 모바일 웹 노치 영역은 `env(safe-area-inset-*)`로 회피, 콘텐츠 침범 금지 |
| 사용자 글자 확대 대응 | 브라우저 200% 확대까지 가로 스크롤 없이 유지 |
| Orientation | 모바일 세로 우선, 가로도 깨지지 않게 흐름 유지 |
| 범위 밖 동작 | 360px 미만은 가로 스크롤 허용 |
```

### 모바일 전용 (네이티브 — 폼팩터는 동일 축, OS·도구만 다름)

> 같은 "모바일 전용" 폼팩터는 iOS·Android에서 동일한 레이아웃 관심사를 공유한다. 시스템 예약 영역·글자 확대의 *플랫폼별 메커니즘*만 다르고 결정 항목은 같다. **시스템 예약 영역 처리는 선택이 아니라 필수다** — 안 하면(특히 Android API 35+·iOS edge-to-edge 기본) 요소가 상태바·내비바·노치 뒤에 가려지고, 가려진 요소는 UI 자동화의 탭이 빗나가 검증이 실패한다(실측 확인).

```markdown
## 디자인 정책 - 화면 레이아웃

| 항목 | 정책 |
|------|------|
| 지원 폼팩터 | 모바일 전용 (태블릿은 호환 모드 — 전용 레이아웃 미제공) |
| 폼팩터별 기준 크기 | 기준 기기군 = 모바일 폰 (iOS=iPhone, Android=일반 폰 폭 — 최소~최신 화면 폭 대응) |
| 시스템 예약 영역 | 콘텐츠는 safe area 내부, 스크롤 배경만 edge-to-edge (iOS=노치·홈 인디케이터·다이내믹 아일랜드 / Android=상태바·제스처 내비게이션·디스플레이 컷아웃 `WindowInsets` 회피). edge-to-edge가 기본이므로 인셋 미처리 시 요소 가림 발생 |
| 사용자 글자 확대 대응 | 기본~최대 글자 크기까지 레이아웃 유지, 초대형은 클램프 (iOS=Dynamic Type / Android=Font Scale `sp` 단위) |
| Orientation | 세로 고정 |
| 범위 밖 동작 | 미지원 OS 버전은 설치 차단 (배포 타깃 최소 버전으로 강제) |
```
