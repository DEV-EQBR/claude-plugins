# 타임존 처리

> 버전: v1.1.0
> 최종 수정: 2026-06-09
> 적용 조건: 다음 중 하나라도 해당하면 적용
> - 도메인 엔티티(`entities.md`)에 시간/날짜 속성이 있는 경우
> - 화면(`screens.md`)에 시간/날짜 표시 요소가 있는 경우
> - 시간 기반 시스템 프로세스(만료 처리, 스케줄링 등)가 있는 경우

---

## 메타-규칙

시간 데이터는 **저장 시간대**와 **표시 시간대**를 분리한다. 저장은 시간대 정보가 명시된 단일 표준으로 통일하고, 표시는 사용자 컨텍스트의 로컬 시간으로 변환한다. 변환은 시스템 내 단일 경계 위치에서만 수행한다.

서버는 시간 값을 변환·재해석하지 않고 저장 표준 그대로 통과시킨다(pass-through, 읽기·쓰기 양방향). 변환 책임은 프론트엔드 단일 경계에 둔다. 생성·갱신 요청 시 클라이언트는 저장 표준 시간대로 변환하여 전송하고, 서버는 수신 값이 표준 시간대인지 검증한 뒤 저장한다.

## 결정 항목

이 컨벤션을 따르려면 다음 항목들을 결정하여 상세설계서에 명시해야 한다.

| 결정 항목 | 명시 위치 | 형태 / 제약 |
|----------|---------|-----------|
| 저장 시간대 표준 | `system.md` / 시간 데이터 처리 섹션 | 단일 표준 시간대(보통 UTC). 시간대 정보가 명시된 저장 형식 (예: ISO 8601 with `Z`, `TIMESTAMP WITH TIME ZONE`) |
| 표시 시간대 결정 방식 | `system.md` / 시간 데이터 처리 섹션 | 셋 중 택일: ① 사용자 브라우저/디바이스 시간대 ② 사용자 프로필 설정 ③ 프로젝트 고정 시간대(예: KST) |
| 변환 경계 위치 | `system.md` / 시간 데이터 처리 섹션 | 둘 중 택일: ① 백엔드 응답 시점에 변환 후 응답 ② 백엔드는 UTC로 응답, 프론트엔드 렌더링 직전 변환 |
| 생성·갱신 요청 시간대 정책 | `system.md` / 시간 데이터 처리 섹션 | 클라이언트가 저장 표준 시간대(UTC, ISO 8601 + `Z`)로 변환하여 전송. 서버는 수신 값이 표준 시간대(오프셋 0)인지 검증 후 저장, 불일치 시 4xx 거부. 서버는 시간 값을 재해석·변환하지 않음 |

> 표시 형식 (`yyyy.MM.dd HH:mm` 등 포맷 문자열) 은 본 컨벤션이 아닌 `service/frontend/date-display.md` 에서 별도 결정.

## 상세설계 작성 예

`system.md` 에 다음 섹션을 추가한다.

```markdown
## 시간 데이터 처리

| 항목 | 정책 |
|------|------|
| 저장 시간대 | UTC (DB 컬럼은 `TIMESTAMP WITH TIME ZONE`, 직렬화 시 ISO 8601 + `Z`) |
| 표시 시간대 결정 | 사용자 브라우저 시간대 (`Intl.DateTimeFormat` 기준) |
| 변환 경계 | 백엔드는 UTC로 응답, 프론트엔드 렌더링 직전에 변환 |
| 생성·갱신 요청 | 클라이언트가 UTC(ISO 8601 + `Z`)로 변환하여 전송. 서버는 오프셋 0 여부 검증 후 저장, 불일치 시 4xx. 서버는 시간 값을 변환하지 않고 pass-through |
```
