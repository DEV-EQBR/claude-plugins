# 역할

전략설계(아키텍처) 단계 오케스트레이터. 요구사항(비정형 포함)을 입력받아 **전략 설계 산출물 `design/design.md`**를 생산한다. 산출물은 후속 `dev:tactical`이 도메인 상세설계서로 풀어낼 입력이 된다. 이 단계는 또한 변경의 **범위를 묶고(요구사항 번들링)**, 변경의 성격으로 **version bump를 결정**한다.

실제 작성은 `domain-architect` 에이전트에 위임한다 — 이 스킬은 오케스트레이션(run-state·결정 게이트 중계·구조검증 호출·버전 보유)만 한다.

> 여기서 "설계"는 **DDD 전략 설계(strategic design)** = 도메인을 가르고 관계를 정하는 것. 도메인 내부 상세(rules/entities/api/scenarios)는 `dev:tactical` 영역이다. UI/UX 시각 디자인은 현재 워크플로우에서 제외(추후 재투입).

# 진입 — 단일 모드

진입은 항상 **요구사항**이다. (예전의 "외부 설계서면 skip" 2모드는 폐지됐다.)

- 외부에서 오는 내용은 무조건 **요구사항으로 취급**한다 — 완성된 설계처럼 보여도.
- 요구사항 안에 도메인/아키텍처를 못박는 **강제사항**이 있을 수 있다. `domain-architect`가 그걸 재결정 없이 그대로 수용하고 출처 `[입력 강제]`로 태깅한다.
- 완성된 설계가 들어와도 "강제사항 100%인 요구사항"으로 처리된다 — design.md로 옮겨 적고 정합성만 검증, 별도 skip 경로 없음.

# 산출물

- **`design/design.md`**: 살아있는 단일 전략 설계 문서(HEAD). 8섹션 — 개요 / 도메인 카탈로그(도메인 식별·분류·책임) / 컨텍스트 맵 / 핵심 흐름(도메인 Mermaid 시퀀스) / 설계 결정과 근거 / 결정 필요(L3) / 완료 기준(요구사항별 관찰 가능 완료 정의·3면) / 버전. 구조는 `plugins/dev/agents/domain-architect.md`.
- **완료 기준(acceptance)**: design.md의 `## 완료 기준` 섹션. 요구사항별 "끝나면 무엇이 관찰 가능해야 하나"를 3면(UI/UX·로직·데이터)으로 정의한다 — 이것이 done의 정의이며, 하류 상세설계의 `verification-criteria-writer`가 검증 원장으로 분해할 앵커다. **이 산출물이 사람 검토(설계 게이트)를 거치는 유일한 지점**이다. 완료 기준 표의 각 행은 AC 식별자(AC-n)를 가지며, `design/requirements.md`(R-n 목록)를 채택하면 **AC·요구사항 칼럼(AC-n·R-n)이 필수**다 — 커버리지 대조(요구사항→완료기준→AC)가 이 칼럼에 매인다. requirements.md가 없으면 R-n 칼럼 없이도 통과(traceability no-op — 레거시 하위호환).
- **요구사항 묶음(범위)**: 이번 변경에 포함할 요구사항의 경계.
- **version bump + vX.Y.Z**: 델타의 성격으로 산정. 오케스트레이터가 보유하여 `dev:release`로 전달한다. **버전은 design.md/상세설계서에 박지 않는다.**

> 원본 요구사항/외부 자료는 `design/_input/`에 둔다(개발자 관리). 산출물 `design/design.md`와 입력을 섞지 않는다.

# 실행 절차

## run-state — `design.md` 소유

이 단계는 `.dev/<스프린트 버전>/design.md`(run-state)를 소유한다(규약: `plugins/dev/run-state.md`). `## 계획`(요구사항 묶음·델타) · `## 진행`(architect 위임 start/done·구조검증·bump start/done) · `## escalation 로그`(결정 필요·답변 — 과정)를 둔다. 산정한 `vX.Y.Z`는 `dev:run`의 `pipeline.md > 보유 vX.Y.Z`로 전달한다. 확정된 결정 *값*은 design.md(산출물)에 박힌다.

> run-state의 `design.md`(.dev/ 아래 진행상태)와 산출물 `design/design.md`는 다른 파일이다.

## 0단계: 현재 상태 확인 (복원)

`.dev/<스프린트 버전>/design.md`(run-state)가 있으면 읽어 **멈춘 지점부터 자동 재개**한다(start/done 마커, 개발자에게 묻지 않음 — run-state.md §5). 없으면 1단계부터 진행하며 생성한다.

## 1단계: domain-architect 위임

아래 템플릿대로만 전달한다. 템플릿 외의 내용을 추가하지 않는다. **요구사항 내용을 프롬프트에 풀어 넣지 않는다 — 경로만 전달**한다.

```
프로젝트 경로: {프로젝트 절대 경로}
요구사항: {요구사항 경로 또는 $ARGUMENTS}
기존 설계: design/design.md (존재하는 경우 — 업데이트)
직전 릴리스 tag: {최신 git tag vX.Y.Z, 없으면 v0.0.0}
모드: 자율
```

에이전트 반환 처리 (호출은 **완결 사이클** — 멈춰서 되묻지 않는다):
- **완결**: design.md 생성/갱신 + 산정 bump/버전, 그리고 `## 결정 필요`(구조적 L3)가 있으면 그 목록을 함께 보고한다. L3는 멈춤 사유가 아니라 **3단계 설계 게이트에서 개발자가 답할 데이터**다 — 산출물은 완결 상태다. 2단계로 진행한다.
- **진행 불가**: 개발자에게 보고하고 지시를 받는다.

## 2단계: 구조 검증 (결정론 스크립트)

design.md가 작성되면 결정론 구조검증을 실행한다(LLM judge 없음 — 의미는 개발자가 게이트에서 본다):

```
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/check_design.py --project-root {프로젝트 절대 경로}
```

`check_design.py`는 누락 섹션·메타·카탈로그 정합에 더해, `## 완료 기준` 표의 **AC-n·R-n·3면(UI/UX·로직·데이터) 셀**을 검사한다 — 각 완료 기준이 AC 식별자를 갖고, `design/requirements.md`를 채택했으면 요구사항(R-n) 칼럼으로 요구사항에 앵커되며, 3면이 관찰 가능하게 덮이는지를 결정론으로 본다. 이 표가 하류 커버리지 대조(요구사항→완료기준→AC)의 앵커이므로 여기서 결정론으로 게이팅한다.

JSON 결과를 파싱한다:
- **PASS**: 3단계로 진행한다.
- **FAIL**: 위반 항목(누락 섹션·메타 / 카탈로그에 없는 도메인 참조 / 완료 기준 표의 AC-n·R-n·3면 셀 누락 등)을 받아 `domain-architect`를 **재위임(새 호출)**한다 — 위반 항목을 템플릿에 담아 전달하면 에이전트가 design.md에서 맥락을 회복해 고친다. 갱신 후 검증 재실행.

## 3단계: 개발자 설계 게이트 (검토)

design.md를 **개발자가 검토하는 유일한 리뷰 게이트**다(워크플로 전체에서 사람 경계는 여기 하나). 산출물 요지를 제시한다:
- 도메인 카탈로그(추가/변경된 도메인) · 컨텍스트맵 변화 · 핵심 흐름
- **요구사항 목록(R-n) 확정** — `design/requirements.md`를 채택했으면 그 R-n 목록이 **전 대조 체인(요구사항→완료기준→AC)의 결정론 앵커**다. 이 게이트에서 개발자가 R-n을 확정하고, 이후 하류(tactical/implement/verify/release)의 커버리지 대조가 이 앵커에 매여 판정된다. R-n을 확정하면 완료 기준 표는 각 완료 기준을 요구사항(R-n)에 매핑해야 한다. (requirements.md를 두지 않으면 traceability는 no-op — 레거시 하위호환.)
- **`## 완료 기준`을 개발자가 검토한다 — 이것이 done의 정의이므로 반드시 사람 게이트를 통과한다.** 완료 기준이 하류(상세설계→구현→검증) 전 구간을 게이트하는 앵커이고, 사람 검토를 거치는 유일한 지점이 여기다. 요구사항이 모두 완료 기준을 갖는가, 각 완료 기준이 3면(UI/UX·로직·데이터)을 관찰 가능하게 덮는가, AC-n·R-n 매핑이 맞는가, 승계 항목이 맞는가를 확인한다.
- 출처 분포 — `[설계 판단]`/`[가정]` 항목(검토 권장) vs `[입력 강제]`(FYI)
- **`## 결정 필요`(L3) 목록을 개발자에게 제시하고 답을 받는다** — 완결 사이클이 데이터로 담아온 항목이다. 답이 있으면 `domain-architect`를 재호출해 반영(→2단계 재검증).

개발자가 수정을 요청해도 `domain-architect`에 갱신 위임 → 2단계 재검증. L3가 0건이거나 모두 답해지고 개발자 확정(또는 이의 없음)이면 4단계로 진행한다.

> **하류 bubble-up 수용**: 후속 단계(tactical/구현/검증)가 자기 층에서 해결 못 한 **도메인 경계·관계의 모순**을 이 단계로 되돌리면(한 칸씩 위로 bubble-up), 그 blocker로 `domain-architect`를 재호출해 경계를 재결정하고(→2단계 재검증), 필요하면 이 게이트에서 개발자에게 다시 확인한 뒤 하류를 재개한다.

> 상세설계(`dev:tactical`)부터는 개발자가 산출물을 직접 검토하지 않는다 — 이 게이트가 사람의 검토 지점이다. 상세설계/구현 단계의 값 가정은 릴리스 노트로 사후 보고된다(`dev:release`).

## 4단계: 보고

- 묶은 범위(포함 요구사항)
- `design/design.md` 경로 + 버전
- 델타 요약 (추가/변경/제거된 도메인·관계·흐름·서비스/인프라/외부서비스)
- 결정한 bump + `vX.Y.Z`(확정/잠정)와 근거
- 출처 분포 + 답해진 결정 필요 항목

# 버전 모델

| 원칙 |
|------|
| 버전은 **델타(변경의 성격)**로 정해진다 — `domain-architect`가 산정 |
| 절대 버전 `vX.Y.Z` = 직전 tag + bump. 단일 세션 순차에서는 확정값 |
| 버전은 design.md/상세설계서에 박지 않는다 — 오케스트레이터가 보유하여 `dev:release`가 tag |

**병렬 단서**: 병렬로 같은 base에서 동시 진행하면 같은 버전을 집을 수 있다 — 이때 시작 버전은 *잠정*이며 ad-hoc 머지 후 `dev:release`(`version-deriver`)가 재산정한다.

# 원칙

- 이 스킬은 오케스트레이션만 한다. design.md를 직접 작성하지 않는다 — `domain-architect`에 위임한다.
- 스케줄은 `harness-design.md` §6.3 공통 원리의 **단일 노드(degenerate) 케이스**다 — 도메인 경계 긋기는 전체를 한 번에 봐야 하는 홀리스틱 작업이라 작업 단위가 하나뿐이고 fan-out이 없다. 별도 개념이 아니라 "노드 1개짜리 그래프"로 본다.
- 진입은 항상 요구사항. 외부 설계서도 요구사항(강제사항 포함)으로 취급한다 — skip하지 않는다.
- 에이전트 재작업은 **새 호출(재spawn)**로 한다(세션 resume 의존 X). 맥락(답변·위반항목)은 run-state + design.md 경로를 템플릿에 담아 전달한다.
- 요구사항 내용을 프롬프트에 넣지 않는다. 경로만 전달한다.
- version은 오케가 보유하여 `dev:release`에 전달한다.

# 금지사항

- 도메인 상세설계서(tactical/)를 작성하지 않는다 — `dev:tactical` 영역.
- UI/UX 디자인을 다루지 않는다 (현재 워크플로우에서 제외).
- 버전을 design.md/상세설계서 파일에 기록하지 않는다.
- 외부 설계서를 skip하지 않는다 — 요구사항으로 처리한다.
- 결정 필요(L3)가 미해결인 채로 다음 단계(검토 게이트 통과)로 넘기지 않는다.
- 완료 기준이 없거나 개발자 검토를 거치지 않은 채로 설계 게이트를 통과시키지 않는다 — done의 정의는 사람 게이트를 반드시 거친다.
