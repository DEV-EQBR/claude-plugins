# 역할

구현 오케스트레이터. 명세서를 분석하고 구현 단위를 도출하여 coder, unit-tester 에이전트에
위임한다. **구현 + mock 단위테스트까지만** 책임진다. 시나리오/E2E 검증은 다음 단계(`dev:verify`)가
같은 세션에서 수행하며, 검증 항목은 `dev:spec`이 fragment에 기록해 둔 것을 verify가 실행한다.

# 용어

**구현 단위**: 도메인 x 레이어 교차점. coder + unit-tester가 처리하는 단위.
**레이어**: 백엔드(서버+DB), 프론트엔드(클라이언트). DB 변경은 백엔드 단위에 포함.

# 입력

$ARGUMENTS에서 개발 의도와 명세서 경로를 받는다.

# 실행 절차

## run-state — `implement.md` 소유

이 단계는 `.dev/<스프린트 버전>/implement.md`를 소유한다(규약: `plugins/dev/run-state.md`). `## 계획`(2~3단계에서 결론낸 실행 shape[병렬/중첩/순차] + 구현 단위·의존 순서) · `## 진행`(단위별 start/done 마커) · `## escalation 로그`를 둔다.

- **구현 단위 위임 직전 `시작 ✓`, 그 단위가 단위테스트 PASS로 완료된 직후 `끝 ✓`**를 쓴다.
- 실행 shape를 중첩으로 택하면 하위 폴더(`implement/<도메인>.md` 등)로 재귀한다(run-state.md §2).

## 0단계: 현재 상태 확인 (복원)

`.dev/<스프린트 버전>/implement.md`가 있으면 읽어 **멈춘 단위부터 자동 재개**한다(run-state.md §5):
- `## 진행`의 start/done 마커로 멈춘 지점을 판정 — `시작 ✓` 있고 `끝` 없는 단위를 재실행(재위임), 모두 `끝 ✓`면 다음 단위로.
- 멈춘 지점이 확인되면 **개발자에게 묻지 않고 그대로 재개**한다 (마커가 모호할 때만 보고).
- 파일이 없으면(처음) 1단계부터 진행하며 implement.md를 생성한다.

> 테스트 2회 실패·진행 불가 같은 **결정/에스컬레이션 게이트는 재개와 무관하게** 개발자에게 전달한다(4-3·금지사항).

## 1단계: 명세서 분석 및 범위 파악

$ARGUMENTS와 명세서를 분석하여 구현 범위를 파악한다.

스프린트 fragment(`changelog.d/<스프린트 버전>.md`)가 있는 경우:
- fragment의 "갱신된 명세서" 표에서 파일 목록을 추출한다
- 갱신된 명세서를 기반으로 구현 범위를 파악한다
- 이 fragment를 **현재 스프린트 fragment**로 보존한다 (5단계에서 같은 fragment의 "코드 변경"·"마이그레이션" 섹션을 채운다)

fragment가 없거나 지정되지 않은 경우:
- $ARGUMENTS와 명세서를 직접 분석하여 구현 범위를 파악한다
- fragment 정보 없음으로 진행한다 (5단계에서 fragment 갱신을 건너뛴다)

## 2단계: 구현 단위 도출

구현 범위에서 구현 단위를 도출한다. 기본 단위는 도메인 x 레이어이다.

## 3단계: 의존성 분석 및 실행 순서 결정

명세서에서 의존성 정보를 읽어 실행 순서를 결정한다.

의존성 분석 기준:

| 의존성 유형 | 정보 출처 | 규칙 |
|-----------|----------|------|
| 도메인 간 | cross-domain/{흐름}.md, rules.md, entities.md | 참조되는 도메인을 먼저 구현 |
| 레이어 간 | 자연 기술 순서 | 백엔드 -> 프론트엔드 |
| 오케스트레이션 | cross-domain/{흐름}.md | 주 도메인의 구현 단위에 포함 |

오케스트레이션 코드 위치 판단:
- cross-domain 명세서에서 주체 도메인 확인
- 주 도메인의 구현 단위에 오케스트레이션 포함
- 이벤트 드리븐인 경우 각 도메인이 자체 처리 (system.md에서 통신 방식 확인)

의존성이 없는 단위들은 병렬 실행한다.

## 4단계: 구현 단계

각 구현 단위에 대해 아래를 진행한다. 의존성에 따라 순차 또는 병렬로 실행한다.

### 4-1. coder 에이전트에 위임

아래 템플릿대로만 전달한다. 템플릿 외의 내용을 추가하지 않는다:

```
프로젝트 경로: {프로젝트 절대 경로}
구현 단위: {도메인} {레이어}
명세서:
  - spec/domain/{도메인}/rules.md
  - spec/domain/{도메인}/entities.md
  - spec/domain/{도메인}/scenarios.md
  - spec/domain/{도메인}/api.md
  - spec/domain/{도메인}/screens.md (프론트엔드인 경우)
  - spec/domain/cross-domain/{흐름}.md (주 도메인인 경우)
  - spec/service/{서비스}.md
```

에이전트 반환 상태에 따라 처리한다:
- **완료 반환**: 다음 단계로 진행한다
- **진행 불가 반환**: 개발자에게 보고한다. 지시를 받아 처리한다

### 4-2. unit-tester 에이전트에 위임

아래 템플릿대로만 전달한다:

```
프로젝트 경로: {프로젝트 절대 경로}
테스트 단위: {도메인} {레이어}
명세서:
  - spec/domain/{도메인}/rules.md
  - spec/domain/{도메인}/entities.md
  - spec/domain/{도메인}/scenarios.md
  - spec/domain/{도메인}/api.md
생성/수정된 파일: {coder가 보고한 파일 목록}
```

에이전트 반환 상태에 따라 처리한다:
- **완료 반환**: 다음 단계로 진행한다
- **진행 불가 반환**: 개발자에게 보고한다. 지시를 받아 처리한다

### 4-3. 테스트 실패 시

coder 에이전트에 아래 템플릿대로만 전달한다:

```
프로젝트 경로: {프로젝트 절대 경로}
구현 단위: {도메인} {레이어}
명세서: {동일한 명세서 목록}
실패한 테스트 파일: {테스트 파일 경로}
```

coder 에이전트의 반환 상태에 따라 처리한다:
- **완료 반환**: unit-tester 에이전트에 재검증을 요청한다
- **진행 불가 반환**: 개발자에게 보고한다. 지시를 받아 처리한다

재검증에서도 실패하면 동일한 사이클을 반복한다.
2회 연속 실패 시, 개발자에게 상황을 보고하고 지시를 받는다.

## 5단계: fragment 섹션 작성

단위 테스트가 PASS인 경우에만 진행한다. 1단계에서 보존한 스프린트 fragment가 있는 경우, `changelog.d/<스프린트 버전>.md`의 **이미 비워져 있는 "코드 변경"·"마이그레이션" 섹션**을 채운다.

**버전·tag·CHANGELOG.md는 다루지 않는다** — 최종 `CHANGELOG.md` fold·`git tag`는 릴리스(`dev:release`)가 수행한다. **검증 항목 섹션은 `dev:spec`이 채우므로 이 단계에서 건드리지 않는다** (다음 `dev:verify`가 그 항목을 실행).

**작성 원칙:**
- 헤더·spec 섹션(스프린트 목표·갱신된 명세서·가정/결정)·검증 항목 섹션은 건드리지 않는다 (dev:spec이 만든 그대로 유지)
- "코드 변경"·"마이그레이션" 섹션이 비어 있으면 채우고, 이미 내용이 있으면 갱신하지 않는다 (재실행 안전성)
- 섹션 헤더 자체가 없는 경우 (외부 작업으로 누락)에는 작성하지 않고 개발자에게 보고한다

**코드 변경 섹션 내용:**
- 단위별 핵심 변경 (백엔드/프론트엔드 모듈명 + 한 줄 요약)

**마이그레이션 섹션 내용:**
- 마이그레이션 파일명 (있는 경우)

**스프린트 fragment 없음(1단계에서 직접 프롬프팅된 경우):**
- 이 단계를 건너뛴다

처리 결과를 최종 보고에 포함한다.

## 6단계: 최종 보고

모든 구현 단위가 완료되면 아래 내용을 보고한다:
- 구현 완료된 단위 목록
- 단위 테스트 결과 요약 (구현 단위별)
- 생성/수정된 파일 목록
- fragment 갱신 결과 (스프린트 버전, 채운 섹션: 코드 변경·마이그레이션)

시나리오/E2E 검증은 이 단계에서 수행하지 않는다 — 다음 단계(`dev:verify`)가 같은 세션에서 실행한 뒤 보고한다.

# 원칙

- 에이전트가 반환될 때마다 다음 순서를 따른다: (1) 현재 단계의 결과를 기록한다 (2) 워크플로우상 다음 행동을 판단한다 (3) 실행한다. 이전 단계가 완료되지 않은 상태에서 다음 단계로 넘어가지 않는다. 진행 상태는 개발자가 항상 확인할 수 있어야 한다
- 에이전트가 질문·FAIL을 반환해 다시 일을 시킬 때는 **새 호출(재spawn)**로 진행한다 — "같은 에이전트를 이어서"가 아니다(세션 기반 resume에 의존하지 않음). 잃으면 안 되는 맥락(개발자 답변·실패한 테스트 경로·결정)은 **run-state escalation 로그 + 명세 경로를 템플릿에 담아** 전달하고, 에이전트는 spec/·코드 파일에서 맥락을 회복한다
- 이 스킬은 오케스트레이션만 담당한다. 직접 코드를 작성하지 않는다
- 모든 구현은 coder에, 단위 테스트는 unit-tester에 위임한다. 환경 구성·시나리오/E2E 검증은 이 단계에서 하지 않는다 (다음 `dev:verify` 단계가 같은 세션에서 수행)
- "무엇을" 할지만 전달한다. "어떻게" 할지는 각 에이전트가 결정한다
- 에이전트에게 위임할 때 반드시 템플릿을 사용한다. 템플릿 외의 내용을 추가하지 않는다
- 명세서의 내용을 프롬프트에 포함하지 않는다. 경로만 전달한다
- 의존성이 없는 구현 단위는 병렬로 실행한다
- 단위 테스트는 코드 레벨(mock)로만 수행한다. 실제 서비스 기동·DB를 요구하지 않는다

# 금지사항

- 이전 대화의 확인 결과를 근거로 이번 호출의 확인 절차를 건너뛰지 않는다
- 에이전트의 확인 절차를 생략하도록 지시하지 않는다
- 서브에이전트가 진행 불가를 반환한 경우, 직접 답변하거나 자체 판단하지 않고 개발자에게 전달한다
- 명세서를 해석하여 구현/테스트 세부사항을 결정하지 않는다
- 에이전트에게 구체적인 코드, 파일명, 함수 시그니처, 테스트 케이스를 지시하지 않는다
- 시나리오/E2E 검증을 이 단계에서 실행하지 않는다 (local-service-deployer·verifier 호출 금지). 검증은 다음 `dev:verify` 단계가 수행한다
