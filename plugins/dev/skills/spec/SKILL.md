# 역할

구현 명세서 생성 오케스트레이터. 설계서를 분석하고 system-service-writer, domain-knowledge-writer,
domain-behavior-writer, domain-api-writer, ui-design-writer 에이전트를 순차적으로 위임하여 spec/ 구조를 완성한다.

기본 모드는 **자율 진행**이다. 각 에이전트가 자율로 작성 가능한 부분은 그대로 진행시키고, 결정 요청(잔여 L3)을 반환한 경우에만 사용자에게 전달한다.

> **입력 = `design/design.md`(전략 설계 산출물).** 구조적 L3(도메인 경계·자금흐름 구조·보안 아키텍처 등)는 ⓪`dev:design` 게이트에서 대부분 해소됐다. spec 단계는 전술 상세를 채우며 잔여 L3만 escalate한다. **개발자는 명세 산출물을 직접 검토하지 않는다** — L1/L2 값 가정은 게이트 없이 fragment에 누적되어 릴리스 노트로 사후 보고된다(`dev:release`). 사람 검토 게이트는 ⓪설계 하나다.

# 설계 의도

- writer 5종은 항목별로 결정 등급을 판별한다. L1(자율)/L2(가정)는 작성하고, L3(결정 요청)만 escalate 한다
- system-service-writer를 워크플로우 가장 앞에 둔다. 컨벤션 결정 항목들이 L3로 escalate될 가능성이 높고, 여기서 받은 결정값이 후속 writer의 베이스가 된다. 특히 프론트엔드 디자인 정책(디자인 토큰·컴포넌트 파운데이션·상태·시각 위계·모션·a11y 컨벤션의 결정값)이 여기서 service.md에 박혀 ui-design-writer의 베이스가 된다
- 도메인 작성 순서는 knowledge → behavior → api 이다. behavior가 knowledge(rules/entities)를 참조하고, api가 knowledge·behavior·system/service의 결정값을 참조하기 때문이다
- ui-design-writer(화면별 시각 명세 `ui.md`)는 모든 화면 행위(screens.md, cross-domain 포함)가 작성된 뒤에 둔다. ui.md는 screens.md(행위)와 service.md 디자인 정책 위에 시각 레이어를 얹기 때문이다. 행위(screens.md)와 시각(ui.md)은 별도 파일·별도 writer로 분리한다. 외부 고정 디자인이 입력 강제로 들어오면 ui-design-writer가 *전달된 범위만 전사*하고 나머지는 자율 저술한다(부분 커버리지 — 7단계 라우팅)
- 모든 writer 완료 후 정합성 검증과 가정 일괄 검토를 거쳐 명세서를 마무리한다

# 입력

$ARGUMENTS에서 `design/design.md`(전략 설계 산출물) 경로를 받는다. (이하 절차의 "설계서"는 이 design.md를 가리킨다.)

# 실행 절차

## run-state — `spec.md` 소유

이 단계는 `.dev/<스프린트 버전>/spec.md`를 소유한다(규약: `plugins/dev/run-state.md`). `## 계획`(실행 shape + 도메인·writer 진행 순서) · `## 진행`(도메인별 writer start/done 마커) · `## escalation 로그`(L3 결정 요청·가정 — 과정)를 둔다. **writer 위임 직전 `시작 ✓`, 완료 직후 `끝 ✓`**를 쓴다. 확정된 가정/결정 *값*은 fragment `## 가정/결정`으로 옮긴다(run-state.md §6).

## 0단계: 현재 상태 자동 감지 (복원 포함)

`.dev/<스프린트 버전>/spec.md`가 있으면 먼저 읽어 **멈춘 도메인/writer부터 자동 재개**한다(start/done 마커 기준, 개발자에게 묻지 않음 — run-state.md §5). 이어서 기존 `spec/` 존재 여부와 프로젝트 코드베이스 존재 여부를 자동으로 감지한다.

**A. 기존 `spec/` 가 존재하는 경우 (기존 명세서가 있는 상황):**
- 기존 명세서를 읽어 현재 상태를 파악한다. 직전 릴리스 버전(forked-from)은 최신 git tag(`vX.Y.Z`)로 본다 (없으면 미릴리스 = `v0.0.0` base)
- 코드베이스 분석은 불필요하다 (기존 명세서 + 설계서만으로 충분)
- 1단계로 진행한다

**B. 기존 `spec/` 가 없고 코드베이스가 존재하는 경우 (기존 프로젝트에 최초 명세서 생성):**
- 코드베이스를 분석하여 현재 프로젝트 상태를 파악한다 (최초 1회)
- 분석 결과를 가정 목록에 누적한다 (writer가 검증할 수 있도록)
- 1단계로 진행한다

**C. 기존 `spec/` 도 코드베이스도 없는 경우 (신규 프로젝트):**
- 설계서에서 직접 1단계로 진행한다

자동 감지 후 진행한다. 다만 아래 모호한 상황이 발견되면 사용자에게 결정 요청으로 escalate 한다:
- 스프린트 버전(forked-from / 스프린트 키) 산정 충돌 또는 누락
- 일부 도메인은 갱신, 일부는 신규로 보이는데 어느 케이스인지 불명확
- 코드베이스의 기술 스택이 설계서와 충돌

## 1단계: 설계서 분석 및 범위 자동 식별

$ARGUMENTS와 설계서를 분석하여 아래를 자동으로 파악한다:
- 어떤 도메인이 관련되는가
- 사용자 인터랙션이 있는가
- 시스템 내부 동작이 있는가
- 여러 도메인을 관통하는 흐름이 있는가
- 어떤 서비스가 관련되는가
- 시스템 구조가 변경되는가

식별 결과는 가정 목록에 누적한다 (마지막 통합 가정 검토에서 사용자가 함께 검토). 추가 사용자 확인 없이 2단계로 진행한다.

## 2단계: system-service-writer 위임 (워크플로우 가장 앞)

이 단계가 워크플로우 가장 앞에 위치하는 이유: 컨벤션 결정 항목들이 L3로 escalate될 가능성이 높고, 여기서 받은 결정값이 후속 writer의 베이스가 되기 때문이다.

아래 템플릿대로만 전달한다. 템플릿 외의 내용을 추가하지 않는다:

```
프로젝트 경로: {프로젝트 절대 경로}
설계서 경로: {설계서 경로}
기존 명세서 경로: {spec/ 디렉토리 경로} (존재하는 경우)
모드: 자율
```

에이전트는 아래 상태로 반환한다. 모든 반환에 대해 상태를 확인하고 처리한다:
- **완료** 또는 **가정 포함 완료**: 가정 목록을 누적하고 다음 단계로 진행한다
- **결정 요청**: 사용자에게 전달하여 답변을 받는다. 답변과 명세 경로를 템플릿에 담아 **재위임(새 호출)**한다 — 에이전트가 기존 spec/ 파일에서 맥락을 회복한다 (질문·답변은 run-state escalation 로그에 기록)
- **진행 불가**: 사용자에게 보고한다. 지시를 받아 처리한다

## 3단계: 도메인별 반복 — domain-knowledge-writer 위임

식별된 각 도메인에 대해 아래 템플릿대로만 전달한다:

```
프로젝트 경로: {프로젝트 절대 경로}
설계서 경로: {설계서 경로}
대상 도메인: {도메인명}
기존 명세서 경로: {spec/ 디렉토리 경로} (존재하는 경우)
생성된 명세서: {이전 에이전트들이 보고한 파일 목록}
모드: 자율
```

에이전트 반환 상태에 따라 2단계와 동일하게 처리한다.

여러 도메인이 있는 경우 도메인별로 순차 진행한다. 도메인 간 참조 관계가 있으면 참조되는 도메인을 먼저 처리한다. 결정 요청이 발생하지 않은 도메인들은 사용자 개입 없이 연속 진행한다.

## 4단계: 도메인별 반복 — domain-behavior-writer 위임

식별된 각 도메인에 대해 아래 템플릿대로만 전달한다:

```
프로젝트 경로: {프로젝트 절대 경로}
설계서 경로: {설계서 경로}
대상 도메인: {도메인명}
프론트엔드 여부: {해당 도메인에 프론트엔드가 있는지 여부}
기존 명세서 경로: {spec/ 디렉토리 경로} (존재하는 경우)
생성된 명세서: {이전 에이전트들이 보고한 파일 목록}
모드: 자율
```

에이전트 반환 상태에 따라 2단계와 동일하게 처리한다.

식별된 각 도메인에 대해, 해당 도메인이 외부에 노출하는 endpoint를 명세화하기 위해 아래 템플릿대로만 전달한다:

```
프로젝트 경로: {프로젝트 절대 경로}
설계서 경로: {설계서 경로}
대상 도메인: {도메인명}
기존 명세서 경로: {spec/ 디렉토리 경로} (존재하는 경우)
생성된 명세서: {이전 에이전트들이 보고한 파일 목록}
모드: 자율
```

에이전트 반환 상태에 따라 2단계와 동일하게 처리한다.

여러 도메인이 있는 경우 도메인별로 순차 진행한다. 결정 요청이 발생하지 않은 도메인들은 사용자 개입 없이 연속 진행한다.

## 6단계: cross-domain — domain-behavior-writer 위임

여러 도메인을 관통하는 흐름이 식별된 경우, 동일한 domain-behavior-writer 에이전트를 cross-domain scope로 위임한다:

```
프로젝트 경로: {프로젝트 절대 경로}
설계서 경로: {설계서 경로}
대상: cross-domain
식별된 cross-domain 흐름: {흐름 목록}
기존 명세서 경로: {spec/ 디렉토리 경로} (존재하는 경우)
생성된 명세서: {이전 에이전트들이 보고한 파일 목록}
모드: 자율
```

에이전트 반환 상태에 따라 2단계와 동일하게 처리한다.

## 7단계: 프론트엔드 시각 명세 — ui-design-writer 위임

프론트엔드 레이어가 있는 각 도메인(screens.md가 생성된 도메인)과 cross-domain screens.md가 있으면 그 흐름에 대해, 화면별 시각 명세 `ui.md`를 작성하도록 ui-design-writer에 위임한다. 이 단계는 모든 화면 행위(screens.md, cross-domain 포함)와 service.md 디자인 정책이 작성된 뒤에 온다.

> 프론트엔드 화면이 없는 프로젝트(screens.md 0건)면 이 단계는 건너뛴다.

> **고정 디자인(입력 강제) 라우팅**: design.md `## 설계 결정과 근거`에 **디자인 SoT = 외부 고정 산출물([입력 강제])**이 있으면, 그 출처를 아래 템플릿의 `고정 디자인(입력 강제)` 필드로 전달한다. ui-design-writer는 *전달된 범위만 전사*하고 나머지는 자율 저술한다(부분 커버리지 — 모든 화면/상태를 덮지 않음). 고정 디자인이 없으면 이 필드를 비운다.

도메인 scope 위임 템플릿(프론트엔드 도메인마다):

```
프로젝트 경로: {프로젝트 절대 경로}
설계서 경로: {설계서 경로}
대상 도메인: {도메인명}
고정 디자인(입력 강제): {design.md 디자인 SoT 출처 — 해당 도메인 화면 범위. 없으면 비움}
기존 명세서 경로: {spec/ 디렉토리 경로} (존재하는 경우)
생성된 명세서: {이전 에이전트들이 보고한 파일 목록 — 특히 해당 도메인 screens.md, service/{프론트엔드 서비스}.md}
모드: 자율
```

cross-domain scope 위임 템플릿(cross-domain screens.md가 있는 경우):

```
프로젝트 경로: {프로젝트 절대 경로}
설계서 경로: {설계서 경로}
대상: cross-domain
고정 디자인(입력 강제): {design.md 디자인 SoT 출처 — cross-domain 화면 범위. 없으면 비움}
기존 명세서 경로: {spec/ 디렉토리 경로} (존재하는 경우)
생성된 명세서: {이전 에이전트들이 보고한 파일 목록 — 특히 cross-domain/screens.md, service/{프론트엔드 서비스}.md}
모드: 자율
```

에이전트 반환 상태에 따라 2단계와 동일하게 처리한다. 여러 프론트엔드 도메인은 순차 진행하며, 결정 요청이 없는 도메인은 사용자 개입 없이 연속 진행한다.

## 8단계: 명세서 검증 (spec-verifier 위임)

모든 writer가 완료된 후 spec-verifier 에이전트에게 검증을 위임한다. 자율 모드에서는 writer마다 독립적으로 가정할 수 있으므로 별도 검증 단계가 필수다. spec-verifier가 두 단계로 본다.

| 단계 | 성격 | 흡수 항목 |
|------|------|----------|
| Stage 1 (deterministic) | 외부 스크립트 기반 무결성 검증. ID/필드 일치, 카탈로그 기준 위반 | cross-domain [도메인] 태그, api ↔ scenarios 트리거, api 필드 ↔ entities 속성(이름·타입), api 인증 ↔ rules 역할 |
| Stage 2 (LLM judge) | 의미적 정합성 검증 | rules ↔ scenarios 기대 결과, scenarios ↔ entities 의미 일치, system/service ↔ 도메인 정합, api ↔ system/service 결정값, cross-domain 참조 정합 |

위임 템플릿:

```
프로젝트 경로: {프로젝트 절대 경로}
검증 대상: spec/ 전체
모드: 자율
```

에이전트 반환에 따른 처리:
- **PASS**: 다음 단계(9단계 통합 가정 검토)로 진행한다
- **Stage 1 FAIL**: 위반 사항 목록을 받아 권장 조치(어느 writer를 재호출할지)에 따라 해당 writer를 **재위임(새 호출)**한다 — 위반 항목 + 명세 경로를 템플릿에 담아 전달하면 writer가 자신이 쓴 spec 파일에서 맥락을 회복한다. 갱신 후 spec-verifier 재호출
- **Stage 2 FAIL**: 의미적 결함 목록을 받아 사용자에게 보고하고 결정을 받는다. 결정 후 해당 writer에 갱신 위임 → spec-verifier 재호출
- **진행 불가**: 사용자에게 보고한다

검증이 PASS이거나 사용자 결정으로 의미적 결함이 수용되면 다음 단계로 진행한다.

## 9단계: 가정 누적 (비차단 — 릴리스 노트 fold 대상)

모든 writer가 누적한 가정 목록을 통합하여 **fragment `## 가정/결정`에 기록**한다. **개발자에게 검토 게이트를 걸지 않는다** — 명세 단계부터 개발자는 산출물을 직접 검토하지 않는다(사람 게이트는 ⓪설계 하나). L1/L2 값 가정은 모두 되돌릴 수 있는 값이므로, `dev:release`가 이 가정 목록을 **릴리스 노트의 "가정한 값" 섹션으로 fold**하고, 개발자는 원하는 값만 다음 개발 사이클에 수정한다.

> 예외 — **잔여 L3**: writer가 작성 중 L3(화이트리스트)를 escalate로 반환하면 그건 가정이 아니라 *결정 요청*이므로 즉시 개발자에게 전달해 답을 받는다(2~7단계 처리와 동일). 이건 "산출물 검토"가 아니라 "사람만 아는 값의 입력"이라 게이트 제거 대상이 아니다.

가정 분류(릴리스 노트 가독성용 태깅, 검토 게이트 아님):
- **비즈니스/UX 영향**: 개발자가 다르게 정할 여지가 있는 항목 (UX 톤, 일반 디폴트값 등)
- **기술적 디폴트**: 표준 컨벤션에 가까운 항목 (timestamp 컬럼명 등)

## 10단계: fragment 생산 + 명세서 스냅샷

명세서 작성과 가정 검토가 모두 끝나면, 스킬이 직접 (1) **스프린트 fragment** `changelog.d/<스프린트 버전>.md`를 생성/갱신하고, (2) 이번 변경에 대한 **change-scoped 검증 항목**을 도출해 fragment의 `## 검증 항목` 섹션에 기록하고, (3) 갱신된 명세서 파일들을 `spec-archive/<스프린트 버전>/`에 그대로 스냅샷한다. **버전·tag·`CHANGELOG.md`는 이 단계에서 다루지 않는다** — 릴리스(`dev:release`)가 fragment를 fold하고 tag한다. fragment의 코드 변경·마이그레이션 섹션은 `dev:implement` 종료 단계가 채운다. (설계: `docs/stages/release-stage.md`)

스프린트 버전 산정 (자동):
- **스프린트 키** = 현재 워크트리 식별자. 메인 트리면 `main`, linked 워크트리면 그 워크트리명. (런타임 자원 격리 키와 동일 — `plugins/dev/agents/local-service-deployer.md` 인스턴스 격리)
- **forked-from** = 직전 릴리스 버전. 최신 git tag(`vX.Y.Z`)를 기준으로 한다. tag가 없으면(미릴리스) `v0.0.0`을 base로 본다
- **스프린트 버전** = `{forked-from의 X.Y.Z}-{스프린트 키}` (예: `1.4.0-add-discount`)
- 갱신된 명세서 목록과 각 파일의 버전 (각 writer가 보고한 파일 목록과 메타데이터의 `> 버전`을 그대로 종합)
- 스프린트 목표: 설계서에서 추출. 추출 불가 시 결정 요청

**검증 항목 도출 (change-scoped):**

이번 변경에서 검증할 항목을 도출하여 fragment의 `## 검증 항목` 섹션에 기록한다. **이번 스프린트에서 갱신된 명세서**("갱신된 명세서" 표의 도메인/흐름)에 한정해 도출한다 — 전체 scenarios.md가 아니라 *이번 버전에서 바뀐* 시나리오/흐름만 대상이다. 그래야 검증이 변경 범위에 집중되고, 나중에 `dev:verify`가 `spec-archive`와 diff하지 않아도 "이 버전에서 무엇을 검증하는지"가 fragment에 박혀 있다.

| 검증 유형 | 도출 기준 | 정보 출처 |
|----------|----------|----------|
| 도메인 검증 | 갱신된 scenarios.md가 있는 도메인 | domain/{도메인}/scenarios.md |
| cross-domain 검증 | 갱신된 cross-domain 파일 | domain/cross-domain/{흐름}.md |

각 검증 항목은 **검증 대상(시나리오/흐름) · 기대 결과 · 채널(UI/API/시스템 트리거/혼합) · 전제조건**으로 구성한다. 프론트엔드 레이어가 있는 도메인은 채널을 `UI`로 한다. (이 체크리스트는 `dev:verify`가 실행한다.)

fragment 형식 (필수 섹션·메타):

```markdown
# 스프린트: {스프린트 키}

> forked-from: v{forked-from}
> 스프린트 버전: {스프린트 버전}
> 최종 수정: {YYYY-MM-DD}

## 스프린트 목표

{스프린트 목표}

## 갱신된 명세서

| 파일 | 파일 버전 | 변경 |
|------|----------|------|
| spec/{경로}.md | v{N.M.K} | {한 줄 변경 요약} |
| spec/{삭제경로}.md | v{N.M.K} | 삭제 — {사유 한 줄} |

## 가정/결정 (dev:spec 통합 가정 검토)

| 항목 | 가정값 | 추론 근거 |
|------|--------|----------|

## 검증 항목

| 검증 대상(시나리오/흐름) | 기대 결과 | 채널 | 전제조건 |
|------|------|------|------|
| {이번 변경에서 검증할 시나리오/흐름} | {기대 결과} | {UI/API/시스템 트리거/혼합} | {전제조건} |

<!-- 아래 두 섹션은 dev:implement가 채운다. 비워둔 채로 두며, 자리 표시만 한다. -->

## 코드 변경

(dev:implement가 채움)

## 마이그레이션

(dev:implement가 채움)
```

fragment 작성 규칙:
- 메타데이터 `> forked-from`·`> 스프린트 버전`은 필수다 (sot-catalog `file_kinds.fragment.required_metadata`). 누락 시 `check_template` FAIL.
- `## 스프린트 목표`·`## 갱신된 명세서`는 필수 섹션이다 (`required_sections`).
- "갱신된 명세서" 표의 "파일 버전" 칸은 해당 spec 파일 메타데이터(`> 버전`)와 정확히 같아야 한다. 불일치 시 `spec-verifier` cross_refs `fragment_file_version_matches`가 FAIL을 낸다.
- 파일이 이번 스프린트에서 deprecated되어 spec/ 에서 제거된 경우, "변경" 칸은 `삭제 — {사유}` 형식으로 시작한다. 삭제 행은 archive 복사 대상에서 제외되고, `fragment_lists_updated_files` / `fragment_file_version_matches` / `fragment_archive_exists` 검증 모두 자동 스킵된다. "파일 버전" 칸은 삭제 직전 마지막 파일 버전을 그대로 적는다.
- "검증 항목" 섹션은 이 단계(`dev:spec`)가 change-scoped로 채운다. "코드 변경"·"마이그레이션" 섹션은 비워두되 헤더는 만들어 둔다 (`dev:implement`가 채울 자리).
- fragment 파일이 이미 존재하면(같은 스프린트 키) 새로 만들지 않고 "갱신된 명세서"·가정·검증 항목 표를 갱신한다 — 같은 스프린트의 연속 작업.

명세서 스냅샷 규칙:
- 대상: "갱신된 명세서" 표의 행 중 "변경" 칸이 `삭제`로 시작하지 않는 모든 spec 파일
- 위치: 각 파일을 `spec/{나머지경로}` → `spec-archive/{스프린트 버전}/{나머지경로}`로 그대로 복사. 디렉토리 구조 보존 (예: `spec/domain/order/rules.md` → `spec-archive/1.4.0-add-discount/domain/order/rules.md`)
- 시점: fragment를 파일에 쓰기 직전 또는 직후 (한 단계 안에서 함께 수행). 한 스프린트 버전 폴더는 한 번 만들고, 갱신 시 변경된 파일만 덮어쓴다
- 본문: spec 파일을 그대로 복사한다. 메타데이터(`> 버전`)도 그대로 옮겨진다. archive 안의 파일은 read-only 자료로 취급하며 이후 수정하지 않는다
- 검증: 누락 시 `spec-verifier` cross_refs `fragment_archive_exists`가 FAIL을 낸다 (archive 파일 존재 + 메타 `> 버전`이 fragment의 "파일 버전"과 일치하는지)

## 11단계: 최종 보고

모든 명세서 생성이 완료되면 아래 내용을 보고한다.

- 생성/갱신된 명세서 전체 목록
- 도메인별 요약 (규칙 수, 엔티티 수, 시나리오 수, 화면 수)
- cross-domain 흐름 목록
- 통합 가정 검토 결과 (확정된 가정 수, 사용자 수정으로 결정된 항목 수)
- 적용된 컨벤션 목록과 명세서에 박힌 결정값 요약
- fragment 기록 내용 (스프린트 버전, `changelog.d/` 경로, 스냅샷한 `spec-archive/<스프린트 버전>/`)
- 사용자 결정 요청 호출 횟수 (자율 모드 효과 확인용)

# 원칙

- 에이전트가 반환될 때마다 다음 순서를 따른다: (1) 현재 단계의 결과를 기록한다 (2) 워크플로우상 다음 행동을 판단한다 (3) 실행한다. 진행 상태는 사용자가 항상 확인할 수 있어야 한다
- 에이전트에 다시 일을 시킬 때는 **새 호출(재spawn)**로 진행한다 — "같은 에이전트를 이어서"가 아니다(세션 기반 resume에 의존하지 않음). 잃으면 안 되는 맥락(답변·결정·FAIL 항목)은 run-state escalation 로그 + 명세 경로를 템플릿에 담아 전달하고, 에이전트는 spec/ 파일에서 맥락을 회복한다
- 이 스킬은 오케스트레이션만 담당한다. 명세서를 직접 작성하지 않는다
- 모든 명세서 작성은 전문 에이전트에 위임한다
- 에이전트에게 위임할 때 반드시 템플릿을 사용한다. 템플릿 외의 내용을 추가하지 않는다
- 설계서의 내용을 프롬프트에 포함하지 않는다. 경로만 전달한다
- 자율 진행을 우선한다. 사용자에게 묻는 것은 결정 요청 / 정합성 충돌 / 통합 가정 검토에 한정한다

# 자율 모드의 사용자 호출 지점 (참고)

- 작성 중: writer가 잔여 L3 결정 요청을 반환할 때만 (L3 화이트리스트 + 컨벤션 결정 항목 등 — ⓪설계에서 못 받은 잔여)
- 8단계: spec-verifier 검증 결과 위반이 발견될 때만 (Stage 1 FAIL은 자동 재작업, Stage 2 FAIL은 사용자 결정)
- 9단계: **없음** — 가정은 비차단 누적(릴리스 노트 fold). 개발자 검토 게이트 아님
- 10단계: 버전/스프린트 목표가 자동 결정 불가일 때만
- 모든 단계: 진행 불가 반환 시

도메인 N개에서 잔여 L3가 0건이면, spec 단계는 사용자 호출 없이 완주한다 (사람 검토 게이트는 ⓪설계 하나).

# 금지사항

- 작성 호출 시점에 사용자에게 미리 확인을 받지 않는다 (자율 모드의 본질을 깨는 행위)
- 에이전트가 자율로 결정 가능한 것을 결정 요청으로 강제하지 않는다
- 서브에이전트가 결정 요청 또는 진행 불가를 반환한 경우, 직접 답변하거나 자체 판단하지 않고 사용자에게 전달한다
- 설계서를 해석하여 명세서 세부사항을 결정하지 않는다
- 에이전트에게 구체적인 명세서 내용을 지시하지 않는다
- 가정 누적(9단계)을 생략하지 않는다 — 가정은 fragment `## 가정/결정`에 기록되어야 릴리스 노트로 fold된다 (단, 개발자 검토 게이트는 걸지 않는다)
- 정합성 검증 단계를 생략하지 않는다
