# 시나리오 간 상태 격리 패턴

검증 시나리오는 서로 간섭하지 않아야 한다(비결정 결과 방지). fresh 검증 상태(deployer가 빈 검증 DB+schema, verifier가 seed 적용) 위에서 아래 중 하나로 격리한다.

## 1. 고유 식별자 (권장 — 가벼움)

각 시나리오가 생성하는 엔티티에 시나리오별 고유 키를 부여한다.

- 이메일/코드/이름 등에 `sc005-<run>-<n>` 같은 접두사
- 재기동·재실행해도 충돌하지 않음
- 비즈니스 엔티티는 실제 API로 생성 (raw INSERT 금지)

## 2. 재시드 (상태 의존 시나리오)

상태 전이(예: 폴링 → 완료)처럼 초기 상태가 중요한 시나리오는 시나리오 직전에 해당 전제를 재시드한다.

- seed.sql의 해당 fixture만 재적용하거나 API로 전제 데이터 구성
- 재기동에 의존한 상태 리셋(예: 인메모리 카운터) 금지 — 재기동 부작용은 비결정적

## 3. 외부 더블 상태

mock/테스트 더블이 호출 횟수 등 내부 상태를 가지면, 시나리오 시작 시 명시적으로 리셋(전용 reset 엔드포인트·재기동이 아닌 호출 기반)한다.

> 핵심: 격리는 "재기동으로 우연히 초기화"에 의존하지 않는다. 고유 식별자 또는 명시적 재시드로 결정론을 보장한다.
