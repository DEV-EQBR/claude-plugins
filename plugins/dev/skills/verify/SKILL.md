# 역할

검증 단계 오케스트레이터. 구현된 코드를 실제로 기동하여 **시나리오/E2E 검증**을 같은 세션에서 수행한다.
`local-service-deployer`로 fresh 검증 환경을 구성하고, `verifier`에 검증을 위임하며, 실패 시 원인을
판별해 재시도/에스컬레이션한다.

검증 항목은 **검증 원장(`tactical/verification-ledger.md`)의 AC 단정**을 실행한다. 원장 v2는 커밋·누적되는
살아있는 아티팩트로, 완료 기준(설계)을 AC 블록마다 **면별 다중 단정 테이블**(`| 면 | 단정 | 채널 | 코더 | verifier | 해소 증거 |`)로
분해해 둔 것이다 — 각 단정 `T<n>`은 절차→관측→기대에 채널 1~5(1 정적구조·2 로직단위·3 계약API·4 정적스크린샷·5 실기기)를 달고 있다.
**verify는 단정을 스스로 도출하지 않는다** — "무엇을 검증하나"(단정·채널)는 설계·상세설계가 소유하고,
verify는 "어떻게 검증하나"(채널 시나리오·assert)와 verifier 칸 갱신만 한다. 원장에 단정이 없거나 검증 불가능하면
만들거나 건너뛰지 말고 ①`dev:tactical`(상세설계) 또는 설계로 **bubble-up**한다 — 채점자가 자기 시험문제를 쓰지 않는다.

이 스킬은 오케스트레이션만 담당한다. 환경 구성은 `local-service-deployer`, 검증 실행은 `verifier`
(실행 절차는 `verify-run` skill)에 위임한다. **실채널 검증을 오케스트레이터가 직접 대신 수행·자가판정하지 않는다**
(아래 원칙).

# 전제

- 구현 + 채널 1~3 단정을 `dev:implement`의 coder가 통과시켜 **원장 코더 칸**에 달성 증거를 기록한 상태다(채널 4·5는 코더=N/A로 verify에 넘어온다). 코더 칸 PASS는 done 근거가 아니다 — done은 verifier 칸 PASS + 해소 증거로만.
- 검증 단정은 `dev:tactical`이 생산해 **검증 원장(`tactical/verification-ledger.md`)**에 AC별 면별 다중 단정으로 기록돼 있다. 원장이 없거나 단정이 비었으면 verify가 도출하지 않고 상세설계/설계로 bubble-up한다.
- 단일 환경에서 수행한다(단일 세션). 병렬 ad-hoc 머지 후 main에서 통합 검증으로 호출할 수도 있다.

# 입력

$ARGUMENTS에서 검증 대상 힌트를 받는다 (선택). 비우면 검증 원장의 AC 단정 전체를 대상으로 한다.

# 실행 절차

## run-state — `verify.md` 소유

이 단계는 `.dev/<스프린트 버전>/verify.md`를 소유한다(규약: `plugins/dev/run-state.md`). `## 계획`(검증할 시나리오/흐름·채널·실행 shape) · `## 진행`(시나리오별 start/done 마커 + 결과) · `## escalation 로그`를 둔다. **검증 단위 위임 직전 `시작 ✓`, 통과 직후 `끝 ✓`**.

> 스케줄은 `harness-design.md` §6.3 공통 원리를 따른다 — **작업 단위 = 검증 단위(AC의 각 면×채널 시나리오)**, **의존 엣지** = 서비스 기동→시나리오, **자원 배타(mutex)** = 호스트 입력 점유 UI 채널 · 단일 런타임 상태. 배타·충돌 없는 검증 단위는 병렬, 그 외(공유 상태를 건드리거나 호스트 입력을 점유하는 것)는 순서. 검증은 별도 개념이 아니라 이 공통 모델의 *자원 제약 있는* 적용이다.

## 0단계: 현재 상태 확인 (복원)

`.dev/<스프린트 버전>/verify.md`가 있으면 읽어 **멈춘 시나리오부터 자동 재개**한다(start/done 마커, 개발자에게 묻지 않음 — run-state.md §5). 단 검증은 durable proof가 없으므로 `시작 ✓` 있고 `끝`(PASS) 미확정인 검증은 **재실행**한다. 파일이 없으면 1단계부터 진행하며 verify.md를 생성한다. **단 호스트 입력 점유형 UI 채널**(2단계 게이트 참조)은 자동 재개 대상에서 제외하고, 진입 전 허가 게이트를 다시 거친다.

## 1단계: 환경 구성

`local-service-deployer` 에이전트에 위임한다 (인스턴스 키 = 현재 워크트리; 메인 트리 = `main`):

```
프로젝트 경로: {프로젝트 절대 경로}
상세설계서: tactical/system.md
```

반환 상태에 따라 처리한다 (deployer는 분석+배포를 **한 완결 사이클**로 끝낸다 — 분석만 하고 멈추는 반환은 없다):
- **배포 완료**: 전체 서비스 기동 중 → 2단계로 진행한다
- **bubble-up (상세설계 blocker)**: 외부 서비스 검증 전략이 system.md에 미선언 등 상세설계 공백 → ①`dev:tactical`(상세설계)으로 한 칸 위 bubble-up
- **setup 누락 (A)**: 실제 연결로 선언된 크레덴셜·접속 정보가 setup(`.env`·시크릿)에 없음 → 그 사실을 개발자에게 알려 setup을 채우게 한다 (mid-cycle 질문이 아니라 upfront 누락 보고)
- **진행 불가**: 개발자에게 보고한다

## 2단계: 검증

`verifier` 에이전트에 검증을 위임한다. 검증 대상은 **검증 원장(`tactical/verification-ledger.md`)의 AC 단정**이며,
각 AC의 면별 다중 단정을 verifier가 **단정마다 그 채널로 구동·assert**한다. 한 면 통과로 다른 면을 추론하지 않는다 — 프론트가 있어도
로직 면(실 API·통합 seam의 실제 판정·상태전이)과 데이터 면(DB·전송 페이로드의 값·형태·계약·정합 실측)을 **따로** 확인한다.
verifier는 **채널 1~2**(정적 구조·순수 로직)를 코더가 이미 돌렸어도 결정론이라 **싸게 재실행**하고(신뢰가 아니라 저비용·gaming 차단),
**채널 3~5**(통합 seam·정적 스크린샷·실기기)에 **독립 판정을 집중**한다 — 코더가 구현 시점에 못 본 실제 통합·렌더·실키 현실.
정적 시각 단정(채널 4)은 앱 정상 실행 렌더 스크린샷 + vision 판독으로 판정하며(보류 아님), 보류는 실기기 상호작용(채널 5)에만 정당하다.
AC는 각 단정의 **verifier 칸이 PASS + 해소 증거**여야 done이다.

위임 템플릿은 도메인 검증·cross-domain 검증 두 유형으로 나눈다:

```
프로젝트 경로: {프로젝트 절대 경로}
검증 유형: {도메인 검증 | cross-domain 검증}
대상: {도메인 또는 cross-domain 흐름명}
검증 원장: tactical/verification-ledger.md (해당 대상의 AC 면별 단정 — `| 면 | 단정 | 채널 | 코더 | verifier | 해소 증거 |`)
상세설계서: {대상의 rules/entities/scenarios/api/screens 경로 (cross-domain은 흐름 + 관련 도메인)}
검증 실행: `verify-run` skill을 사용해 각 단정을 그 채널로 구동·assert (채널 1~2 재실행, 채널 3~5 독립 판정 집중, 채널 4 정적 시각은 렌더 스크린샷+vision으로 판정). 즉흥 셸 금지, 외부 호출 timeout 강제
상태 갱신: 실행 결과(단정별 PASS/FAIL/보류/미결 + 해소 증거)를 원장의 **verifier 칸·해소 증거 칸**에만 기록. 코더 칸·단정 본문·채널은 추가/삭제/재작성 금지
```

verifier가 원장에 없는 단정을 발견하거나 어떤 단정이 검증 불가능하다고 반환하면, 오케스트레이터는 그것을 verify에서
만들지 않고 ①`dev:tactical`(상세설계)/설계로 bubble-up한다(채점자 자기 시험문제 금지).

검증 실행 절차는 `verify-run` skill, 검증 채널 원칙(프론트엔드 도메인 UI 채널 의무 등)은
`plugins/dev/agents/verifier.md`를 따른다. 검증 실행 도구·UI 드라이버는 `tactical/system.md` "검증 환경 컨벤션"의
선언을 따른다 — UI 드라이버는 프론트 런타임으로 갈린다(브라우저=Playwright/browser MCP, 네이티브 iOS=`xcodebuild test`+XCUITest, 네이티브 Android=`./gradlew connectedAndroidTest`+Espresso, 네이티브 macOS=`xcodebuild test -destination 'platform=macOS'`+XCUITest). 미선언 시 디폴트(UI=브라우저 자동화, API=Hurl).

### 호스트 입력 점유 채널 게이트 — 진입 전 개발자 허가

선언된 UI 검증 드라이버가 **호스트 입력 점유형**(검증 중 호스트의 실제 키보드/마우스를 점유하여 개발자가 동시에 다른 작업을 못 함 — 현재 네이티브 macOS XCUITest. 점유 여부 판별은 `local-verification.md`의 UI 드라이버 분류)이면, 그 UI 채널 검증을 `verifier`에 위임하기 **전에** 개발자에게 진행 허가를 받는다.

- 점유 사실 + 영향(검증 동안 호스트 입력을 점유 → 다른 작업 불가) + 예상 대상 시나리오를 알리고 진행 여부를 묻는다.
- **허가** → 그대로 진행한다.
- **거부** → **채널 통째 포괄 면제는 폐지됐다.** 영향받는 **각 단정마다** 개별로 처리한다 — 그 단정을 verifier 칸에 **채널 5(실기기) 보류**로 표시하고, **항목별 승인 토큰 `승인: <개발자>·<날짜>·<사유>`를 해당 단정 해소 증거 칸에 기록**한다(승인 토큰이 없는 단정은 L7 위반으로 잡혀 release가 막힌다). 점유 없는 채널(API·트리거·무점유 UI)부터 진행한다. 단, **채널 4(정적 스크린샷)는 보류가 불가하다** — 정적 렌더 스크린샷은 호스트 입력 점유 없이 앱 정상 실행만으로 얻으므로, 렌더하지 않으면 L6가 그 단정을 미종결로 잡는다. 보류/미결이 남아 있으면 `dev:release`로 진행하지 않으며, 미룬 것은 해소 증거 없이 확정(PASS)으로 뒤집히지 않는다. 보류·승인 토큰은 4단계 보고와 escalation 로그에도 남긴다.
- **무점유 드라이버**(웹 브라우저 자동화·iOS/Android 시뮬레이터·에뮬레이터)는 게이트 없이 진행한다.

허가/보류 결과는 `verify.md` `## 진행`에 기록한다.

## 2.5단계: 경계 대조 게이트 (verify 종결 — 결정론)

verifier가 원장의 verifier 칸·해소 증거 칸을 갱신하면, 커밋된 원장(`tactical/verification-ledger.md`)에 대해 verify 경계 종결 게이트를 실행한다:

```
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/run_all.py --project-root {프로젝트 절대 경로} --boundary verify
```

`--boundary verify`는 앞 층(커버리지 COV + L1~L3 + L5) 위에 **L6·L7**을 누적 검사한다:
- **L6 (verifier 종결)**: 비 N-A 단정은 verifier 칸이 **PASS이거나 승인된 채널 5 보류**여야 한다 — 그 외(미착수 `-`·미종결·미승인 보류)면 위반. 채널 4(정적 스크린샷)는 보류가 정당하지 않으므로 렌더·판정 없이 남으면 L6가 미종결로 잡는다.
- **L7 (보류 승인 토큰)**: 보류는 **채널 5**에만 정당하며, 각 보류 단정은 **항목별 승인 토큰 `승인: <개발자>·<날짜>·<사유>`**를 갖춰야 한다 — 포괄 면제는 개별 단정 토큰을 못 채워 위반으로 잡힌다.

**verify 단계의 닫힘(release로 넘길 자격)은 이 게이트의 위반 목록이 판정한다** — verifier는 자기 칸을 기록하되 스스로 경계를 닫지 못한다. 위반이 있으면 3단계(검증 실패 처리)로 원인별 재시도/에스컬레이션한다. **위반 0이어야 `dev:release`로 진행**하며, 이는 종전의 산문 "PASS 아니면 release 차단"을 결정론 게이트로 경화한 것이다.

## 3단계: 검증 실패 처리 (재시도와 에스컬레이션)

검증 실패가 있으면 오케스트레이터가 원인을 판별하여 자동 해결을 시도한다.

**판별 1순위 = 개발 진단 로그**: 표면 증상으로 바로 아래 표에 분류하기 전에, verifier가 실패 진단에 담아온 **개발 진단 로그 발췌**(FAIL 단정의 `diag:` 라인 — system.md "검증 환경 컨벤션"의 로그 조회 규약, deployer가 `.dev/env-state.md`에 남긴 "개발 진단 로그 조회" 명령)를 먼저 읽어 실패 지점을 국소화한다. 로그가 가리키는 이음매로 아래 표의 분류를 강화한다(예: seam 호출 실패 `diag:...level=error` → 구현 오류·통합 seam / 연결 거부 로그 → 환경 오류). 로그는 진단용이며 판정·PASS 근거가 아니다.

| 판별 신호 | 분류 | 처리 |
|-----------|------|------|
| 코드 실행 중 에러(500/예외/잘못된 응답값), UI 화면이 screens.md와 불일치 | 구현 오류 | `dev:implement`로 해당 단위 재구현 또는 개발자 보고 (코드 수정은 개발자 판단) |
| 연결 거부, 타임아웃, 서비스 미응답 | 환경 오류 | `local-service-deployer`에 재구성 위임 후 재검증 |
| 선언된 채널(특히 UI)로 끝까지 구동 못 함 = **미도달** / 실 통합 seam이 mock으로만 검증됨 / 3면 중 한 면이라도(로직·데이터 포함) 실채널 실측 없이 다른 면 추론으로 통과 처리됨 | 미충족(비PASS) | PASS 아님. 실 런타임/실채널 구동을 재시도하거나, 호스트 점유 등으로 불가하면 **원장에 보류/미결로 남기고 release 차단**(에스컬레이션). mock 단위로 대체하거나 한 면 통과로 다른 면을 PASS 처리 금지 |
| 모션 불변식이 상세설계됐는데 계측 프로브·결정론적 재생 모드가 구현에 없음 / 모션 판정이 프로브 로그 실측이 아니라 소스 대조·프레임 눈 판독에 근거함 | 구현 누락(비PASS) | 프로브·재생 모드는 coder의 동반 구현 의무 — `dev:implement`로 재구현. 소스 대조 기반 모션 PASS는 수락하지 않는다 |
| 상세설계(상세설계) 자체가 어긋남 (구현이 아니라 상세설계가 모순) | 상세설계 모순 (C) | ①`dev:tactical`으로 **bubble-up**(한 칸 위) — 구현이 아니라 상세설계를 고쳐 재검증 |
| 판별 불가 | 그 외 | 개발자에게 보고 |

- 검증은 **최대 3회까지 재시도**한다 (해결 → 재검증 1회를 1시도로 센다)
- **3회 후에도 실패하면 개발자에게 에스컬레이션**한다 — 실패 시나리오, 시도한 해결, 잔존 증상을 정리하여 보고하고 멈춘다

verify 경계 종결 게이트(`run_all.py --boundary verify` — L6·L7)가 **위반 0**이어야 다음 단계(`dev:release`)로 진행할 수 있다.

> **PASS가 아닌 모든 상태(FAIL·미도달·보류·미결·도구 제한·부분 검증)는 release 차단이다.** 이 차단은 산문 규칙이 아니라 2.5단계 결정론 게이트(L6·L7)로 집행된다 — 미종결·미승인 보류가 남으면 게이트가 위반으로 잡는다. AC는 각 단정의 **verifier 칸이 PASS + 해소 증거**여야 done이며(코더 칸 PASS만으로는 done 아님), 한 단정이라도 실채널 실측 없이 남으면 그 AC는 done이 아니다. verifier가 'PASS' 외의 결과를 반환하거나, 어떤 AC의 어떤 단정이라도 *선언된 실 채널로 끝까지 구동돼 확인되지 않았으면* release로 진행하지 않는다. **"조건부 PASS"처럼 미정의 통과 상태를 만들어(또는 verifier가 반환한 그런 상태를 수락해) 진행하지 않는다** — 검증 불가는 PASS가 아니다. UI 채널을 mock 단위로 대체하거나, 로직·데이터 면을 UI 통과로 추론한 것은 PASS 근거가 아니므로 release를 막는다. **호스트 점유 게이트 보류·미도달은 원장에 보류/미결로 남아 release를 차단하며, 확정(PASS)으로 뒤집히지 않는다.**

## 4단계: 최종 보고

검증이 완료되면 아래 내용을 보고한다:
- 검증한 AC 목록과 각 AC의 **면별 단정(`T<n>`) 판정 + 해소 증거** (원장 verifier 칸과 일치). 채널 1~2 재실행분과 채널 3~5 독립 판정분 구분
- 각 단정에 사용한 **검증 채널** (1 정적구조 / 2 로직단위 / 3 계약API / 4 정적스크린샷 / 5 실기기)
- 보류/미결로 원장에 남아 release를 차단하는 항목(호스트 점유 게이트 보류 등)
- 재시도 횟수
- 문제 발견 시 증상과 재현 방법
- **성능 보고** — 서비스 단위 CPU/메모리(전구간 + 시나리오별 avg/peak)와 전구간 시계열 요약(`verify-run` 성능 샘플러 산출). 측정원·간격과 수치 caveat(샘플링 기반, docker MEM은 절대값·MEM%는 Docker VM 분모) 명시. 정책상 미측정이면 그 사실을 명시
- 생성된 테스트 데이터 정리 여부

# 원칙

- 에이전트가 반환될 때마다 다음 순서를 따른다: (1) 결과를 기록한다 (2) 다음 행동을 판단한다 (3) 실행한다. 진행 상태는 개발자가 항상 확인할 수 있어야 한다
- 이 스킬은 오케스트레이션만 담당한다. 환경 구성·검증 실행을 직접 하지 않고 위임한다
- **실채널 검증을 오케스트레이터가 직접 수행·자가판정하지 않는다.** 호스트 점유 UI 허가를 받은 뒤라도 osascript·스크린샷·직접 셸로 대신 검증하지 않는다 — 반드시 `verifier`(+`verify-run`) 위임을 경유해야 verifier의 PASS/FAIL·미도달·mock 금지·3면 규율이 실제로 적용된다. 오케는 위임·상태 조율·bubble-up 판단만 한다
- **원장 갱신은 verifier가 실행한 단정의 결과(PASS/FAIL/보류/미결 + 해소 증거)를 원장 verifier 칸에 기록하고 오케가 조율한다.** verify는 원장의 단정 본문·채널·코더 칸을 추가/삭제/재작성하지 않고 **verifier 칸·해소 증거 칸만 갱신**한다
- "무엇을" 할지만 전달한다. "어떻게" 할지는 각 에이전트가 결정한다. 무엇을 검증하나(단정)는 원장(설계·상세설계 소유)에서 받고, verify는 어떻게(채널·assert)만 결정한다
- 상세설계서·원장의 내용을 프롬프트에 포함하지 않는다. 경로만 전달한다
- 검증 실행은 `verify-run` skill을 통한다 (즉흥 셸 구현 금지, 모든 외부 호출에 timeout)
- **호스트 입력 점유가 필요한 검증은 진입 전 개발자 허가가 있어야 한다.** 허가 없이 호스트 키보드/마우스를 점유하는 채널(네이티브 macOS UI 등)에 진입하지 않는다. 점유 여부는 `system.md` UI 드라이버 선언(`local-verification.md` 분류)으로 판별한다

# 금지사항

- 검증을 3회를 넘겨 무한 재시도하지 않는다. 3회 후 개발자 에스컬레이션한다
- 서브에이전트가 `{blocker}`(상세설계 모순)를 반환하면 자체 판단하지 않고 ①`dev:tactical`(상세설계)으로 bubble-up 한다. setup 누락(크레덴셜)은 개발자에게 누락 보고, 그 외 진행 불가는 개발자에게 보고한다
- **검증 단정을 verify가 스스로 도출하지 않는다.** 원장에 단정이 없거나 어떤 단정이 검증 불가면 만들거나 건너뛰지 않고 `dev:tactical`/설계로 bubble-up 한다. 원장 단정 본문·채널·코더 칸을 verify가 추가/삭제/재작성하지 않는다(verifier 칸·해소 증거 칸만 갱신)
- **실채널 검증을 오케스트레이터가 직접(osascript·스크린샷·직접 셸) 대신 수행·자가판정하지 않는다** — 호스트 점유 허가 후에도 반드시 `verifier`(+`verify-run`) 위임을 경유한다
- 3면 중 한 면(UI/UX·로직·데이터) 통과로 다른 면을 추론해 PASS 처리하지 않는다 — 로직 면(실 API·통합 seam)과 데이터 면(DB·전송 페이로드 실측)은 UI와 무관하게 각자 실채널로 확인한다
- UI 채널 검증을 생략하지 않는다 (프론트엔드 도메인은 UI 채널 의무 — `verifier.md` 원칙)
- UI 채널 검증을 mock 단위 테스트로 대체해 PASS 처리하지 않는다 — 실 런타임 구동이어야 한다 (실 통합 seam은 mock 대체 불가 — `verifier.md`)
- 호스트 입력 점유형 UI 채널(네이티브 macOS 등)을 개발자 허가 없이 진입하지 않는다 (허가 시 진행, 거부 시 보류 — 보류는 생략이 아니며 release 전 해소 필요)
- **"조건부 PASS" 등 미정의 통과 상태를 만들거나 수락해 `dev:release`로 진행하지 않는다** — 미도달·보류·부분·mock 대체는 모두 비PASS = release 차단
- 모션 불변식이 상세설계된 항목을 소스 대조·프레임 눈 판독 기반 PASS로 수락하지 않는다 — 모션 판정은 프로브 로그 실측만 인정하며, 프로브·결정론적 재생 모드 부재는 구현 누락(비PASS)이다 (`verifier.md` / `verify-run` 모션 불변식 실측)
- 검증 통과 없이 `dev:release`로 진행하지 않는다
